Fundamentos de Redes

Semana 1 — Tudo começa aqui: como os computadores conversam entre si

1 O que é uma Rede?

Imagina um mapa com várias cidades. Agora imagina estradas conectando essas cidades. Uma rede de computadores é exatamente isso: computadores (as cidades) conectados por cabos, Wi-Fi ou fibra óptica (as estradas) para trocar informações.

🛣️ Analogia: Cada computador é uma cidade. Os cabos e roteadores são as estradas e cruzamentos. Os dados são os caminhões carregando mercadoria de uma cidade pra outra.

Existem tipos diferentes de redes:

  • LAN — Rede local (seu escritório, sua casa)
  • WAN — Rede ampla (conecta cidades, países)
  • Internet — A maior WAN do mundo: milhões de redes conectadas

Tudo que você faz no dia a dia — abrir um site, puxar dados de uma API, acessar o Grafana — depende de uma rede funcionando direitinho.

💡 No PicPay... nossos microserviços conversam entre si através de redes internas na AWS (VPCs). Quando você acessa um dashboard ou faz um deploy, seus dados trafegam por várias redes até chegar onde precisam.

2 Endereço IP (IPv4 e IPv6)

Pra um caminhão entregar uma encomenda, ele precisa de um endereço. Na rede, todo dispositivo tem um endereço IP — é como o CEP + número da casa do computador.

🏠 Analogia: Seu endereço de casa permite que o carteiro te encontre. O endereço IP permite que outros computadores encontrem o seu.

IPv4 — O formato clássico. São 4 números de 0 a 255:

192.168.1.100    ← IP privado (rede local)
54.231.0.12      ← IP público (acessível pela internet)

O problema? IPv4 tem "só" ~4,3 bilhões de endereços. Com tanta coisa conectada, acabou. Por isso surgiu o IPv6:

2001:0db8:85a3:0000:0000:8a2e:0370:7334

IPv6 tem 340 undecilhões de endereços. Dá pra dar um IP pra cada grão de areia do planeta e ainda sobra.

Na prática do dia a dia, você vai lidar muito mais com IPv4 privado dentro das VPCs e com DNS (que vamos ver já já) pra não precisar decorar números.

💡 No PicPay... usamos IPs privados dentro das VPCs da AWS (ex: 10.0.x.x). Os serviços se encontram por DNS interno, não por IP direto — o que facilita muito quando um serviço muda de máquina.

3 TCP vs UDP

Beleza, os computadores têm endereço. Mas como os dados viajam? Existem dois jeitos principais — dois "protocolos de transporte".

📬 Analogia: TCP é como enviar uma carta registrada — você sabe que chegou, tem confirmação e a ordem é garantida. UDP é como gritar na rua — rápido, mas ninguém garante que todo mundo ouviu.

TCPUDP
Confiável?Sim — garante entregaNão — best effort
OrdemMantém a ordem dos pacotesPode chegar fora de ordem
VelocidadeMais lento (handshake)Mais rápido
Uso típicoHTTP, APIs, banco de dadosStreaming, DNS, jogos online

TCP faz um "handshake" de 3 etapas antes de começar a mandar dados:

Cliente → SYN        → Servidor     "Oi, posso conectar?"
Servidor → SYN-ACK   → Cliente      "Pode sim, tô aqui!"
Cliente → ACK        → Servidor     "Beleza, vamos lá!"

99% do que você vai trabalhar no Command Center usa TCP — porque APIs e bancos precisam de confiabilidade.

💡 No PicPay... nossas APIs REST e conexões com PostgreSQL, Redis e DynamoDB usam TCP. UDP aparece mais em logs (syslog) e em resoluções DNS internas.

4 DNS — Domain Name System

Ninguém quer decorar 54.231.0.12 pra acessar um site. É aí que entra o DNS — ele traduz nomes legíveis (como picpay.com) pra endereços IP.

📞 Analogia: DNS é a lista telefônica da internet. Você procura o nome ("picpay.com"), e ele te dá o número (IP) pra ligar.

Como funciona na prática:

  • Você digita grafana.picpay.com no navegador
  • Seu computador pergunta ao servidor DNS: "qual o IP disso?"
  • O DNS responde: 10.50.3.22
  • Seu navegador conecta nesse IP

Existem tipos de registro DNS que você vai encontrar:

  • A — Nome → IPv4 (o mais comum)
  • AAAA — Nome → IPv6
  • CNAME — Apelido → outro nome DNS
  • MX — Pra onde vão os e-mails
  • TXT — Metadados (verificação, SPF)

💡 No PicPay... usamos o AWS Route 53 como nosso DNS gerenciado. Ele resolve nomes internos dos serviços (service discovery) e também gerencia os domínios públicos. Quando um serviço muda de IP, o Route 53 atualiza o registro automaticamente — zero downtime.

5 Portas de Rede

O IP leva você até o computador certo. Mas um computador roda vários serviços ao mesmo tempo — web, banco de dados, SSH... Como saber pra qual serviço o pacote vai?

🏢 Analogia: O IP é o endereço do prédio. A porta é o número do apartamento. 10.0.1.5:5432 = prédio 10.0.1.5, apartamento 5432 (PostgreSQL).

Portas que você precisa saber de cor:

PortaServiçoQuando você vê
80HTTPSites sem criptografia
443HTTPSSites com TLS (quase tudo hoje)
53DNSResolução de nomes
22SSHAcesso remoto a servidores
5432PostgreSQLBanco de dados relacional
6379RedisCache e dados em memória
9090PrometheusMétricas de monitoramento
3000GrafanaDashboards de observabilidade

O range vai de 0 a 65535. Portas abaixo de 1024 são "well-known" (reservadas pra serviços padrão). Acima disso, é terra livre.

💡 No PicPay... os Security Groups da AWS controlam quais portas ficam abertas pra cada serviço. Se o Grafana não responde, uma das primeiras coisas a checar é: "a porta 3000 tá liberada no Security Group?"

6 Protocolos HTTP e HTTPS

HTTP é a língua que navegadores e APIs usam pra conversar. Quando você acessa um site ou faz um curl, está usando HTTP.

A anatomia de uma requisição:

GET /api/v1/users/133 HTTP/1.1
Host: api.picpay.internal
Authorization: Bearer eyJhbGciOi...
Content-Type: application/json

Métodos HTTP que você vai usar direto:

  • GET — Buscar dados ("me mostra o usuário 123")
  • POST — Criar algo ("cria um novo alerta")
  • PUT — Atualizar inteiro ("substitui esse registro")
  • PATCH — Atualizar parcial ("muda só o nome")
  • DELETE — Remover ("apaga esse alerta")

Status codes que você precisa reconhecer:

  • 200 — OK, deu certo
  • 201 — Criado com sucesso
  • 400 — Request errada (culpa do cliente)
  • 401 — Não autenticado (falta token)
  • 403 — Proibido (sem permissão)
  • 404 — Não encontrado
  • 500 — Erro no servidor (culpa do backend)
  • 502 — Bad Gateway (proxy não conseguiu falar com o backend)
  • 503 — Serviço indisponível (sobrecarregado ou em deploy)

HTTPS = HTTP + TLS (criptografia). Os dados viajam embaralhados — ninguém no caminho consegue ler. Hoje, tudo em produção é HTTPS.

💡 No PicPay... quando você vê um erro 502 ou 503 no alerta, geralmente o problema é entre o load balancer e o serviço backend. Status 5xx = problema no nosso lado. Status 4xx = problema no request do cliente.

7 Load Balancer

Imagine um restaurante com 5 caixas. Se todo mundo for pro caixa 1, vai ter fila enorme e os outros caixas ficam parados. O Load Balancer é o cara que organiza a fila e distribui os clientes pros caixas disponíveis.

Na prática, um Load Balancer recebe todas as requisições e distribui entre várias instâncias do mesmo serviço:

             ┌──→ Instância 1
Usuários → [LB] ──→ Instância 2
             └──→ Instância 3

Algoritmos comuns de distribuição:

  • Round Robin — Um pra cada, na sequência
  • Least Connections — Manda pra quem tá menos ocupado
  • IP Hash — Mesmo cliente vai sempre pro mesmo servidor

O LB também faz health checks — fica pingando as instâncias pra saber se estão vivas. Se uma morre, ele para de mandar tráfego pra ela.

💡 No PicPay... usamos dois tipos de Load Balancer da AWS: o ALB (Application Load Balancer) pra tráfego HTTP/HTTPS (camada 7 — entende URLs, headers, faz roteamento inteligente) e o NLB (Network Load Balancer) pra tráfego TCP/UDP puro (camada 4 — ultra rápido, milhões de requests por segundo). A maioria dos nossos serviços está atrás de um ALB.

8 WAF — Web Application Firewall

O Load Balancer distribui tráfego, mas e se parte desse tráfego for malicioso? É aí que entra o WAF — ele fica na frente, analisando cada requisição antes de deixar passar.

🛡️ Analogia: WAF é o segurança na porta da balada. Ele olha cada pessoa (requisição), checa se tá tudo certo, e barra quem parece suspeito.

O que o WAF bloqueia:

  • SQL Injection — Tentar injetar comandos no banco de dados
  • XSS — Injetar scripts maliciosos em páginas
  • DDoS (camada 7) — Flood de requisições HTTP
  • Bots maliciosos — Scrapers, scanners de vulnerabilidade
  • Requisições com padrões suspeitos — Payloads estranhos, headers manipulados

O WAF trabalha com regras. Cada regra define o que bloquear, o que permitir e o que apenas logar (modo "count" — só observa sem barrar).

💡 No PicPay... usamos o AWS WAF integrado com os ALBs e o CloudFront. No Command Center, quando você vê um pico de requisições 403, pode ser o WAF barrando um ataque. As regras são gerenciadas pelo time de segurança, mas entender o que o WAF faz ajuda você a diagnosticar incidentes.

9 API Gateway e Kong

Imagina que o PicPay tem centenas de microserviços. O app do celular precisa falar com vários deles. Seria um caos se o app tivesse que saber o endereço de cada um, né?

O API Gateway resolve isso: é um ponto único de entrada que recebe todas as chamadas e roteia pro serviço certo.

App → [API Gateway] → /users   → ms-users
                     → /wallet  → ms-wallets
                     → /payment → ms-payments

O que mais o API Gateway faz:

  • Autenticação — Valida tokens antes de chegar no serviço
  • Rate Limiting — Limita quantas requisições por segundo
  • Logging — Registra todas as chamadas (observabilidade)
  • Transformação — Pode modificar headers, paths e payloads
  • Circuit Breaking — Se um serviço tá fora, para de mandar tráfego pra ele

💡 No PicPay... usamos o Kong como nosso API Gateway principal. Ele é open-source, baseado em Nginx e Lua, e gerencia as rotas, plugins de autenticação, rate limit e mais. Quando um alerta menciona "Kong" ou "gateway timeout", o problema pode estar na rota, no plugin ou no serviço de destino. Saber que o Kong existe e o papel dele é essencial pra diagnosticar problemas de comunicação entre serviços.

10 Gateway de Borda e CloudFlare

Antes de qualquer requisição chegar nos nossos servidores, ela passa pela borda da rede — a primeira linha de defesa e otimização.

🌍 Analogia: Se os seus servidores ficam em São Paulo, e um usuário em Manaus acessa o app, a requisição viaja ~2.800 km. O gateway de borda coloca "embaixadas" (pontos de presença) espalhados pelo mundo pra atender mais perto do usuário.

O que o CloudFlare / CDN faz:

  • CDN (Content Delivery Network) — Cacheia conteúdo estático (imagens, JS, CSS) em servidores espalhados pelo mundo
  • Proteção DDoS — Absorve ataques volumétricos antes de chegar na sua infra
  • TLS Termination — Gerencia os certificados HTTPS na borda
  • Firewall de borda — Bloqueia IPs suspeitos e países inteiros se necessário
  • Anycast — A mesma requisição vai automaticamente pro ponto mais próximo

Fluxo completo de uma requisição:

Usuário → CloudFlare (borda) → WAF → ALB → Kong (API GW) → Microserviço

Cada camada tem um papel. Se der problema, saber onde cada uma fica ajuda a isolar a causa.

💡 No PicPay... usamos CloudFlare como gateway de borda pra proteção DDoS e CDN, além do AWS CloudFront como CDN adicional. Quando rola um ataque DDoS, a primeira defesa é na borda — o CloudFlare absorve o tráfego malicioso antes que ele chegue na nossa infra AWS. No CC, alertas de "high traffic" podem ser legítimos (pico de uso) ou ataque — a borda ajuda a diferenciar.
