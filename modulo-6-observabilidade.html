Observabilidade

Métricas, logs e traces — os 3 pilares pra entender o que acontece nos seus sistemas

1 O que é Observabilidade?

Observabilidade é a capacidade de entender o estado interno de um sistema apenas olhando suas saídas (métricas, logs e traces). Não é só "monitorar se está no ar" — é conseguir responder "por que está lento?" ou "onde o erro aconteceu?" sem precisar acessar o servidor.

A diferença entre monitoramento e observabilidade:

AspectoMonitoramentoObservabilidade
Pergunta"Está funcionando?""Por que não está funcionando?"
AbordagemAlertas pré-definidosExploração livre dos dados
FocoDashboards com métricas conhecidasInvestigação de cenários desconhecidos
Quando serveProblemas previsíveisProblemas que você nunca viu antes

🏥 Analogia: Monitoramento é o monitor cardíaco (bip... bip... bip — tá vivo). Observabilidade é a ressonância magnética — você consegue ver dentro do corpo, entender o que cada órgão está fazendo e descobrir problemas que o monitor cardíaco nunca mostraria.

💡 No PicPay... observabilidade é estratégica. Com centenas de microserviços, não dá pra depender de "olhar log no servidor". Usamos Prometheus (métricas), OpenSearch (logs), Jaeger (traces) e Grafana (dashboards) pra ter visibilidade completa de tudo que acontece — do request do cliente até o banco de dados.

2 Os 3 Pilares da Observabilidade

Toda observabilidade se apoia em 3 tipos de dado. Sozinhos, cada um conta parte da história. Juntos, contam a história completa.

PilarO que éExemploFerramenta
MétricasNúmeros que variam ao longo do tempoCPU 78%, latência p99=200ms, 1500 req/sPrometheus, VictoriaMetrics
LogsEventos textuais com timestamp"2026-03-25 ERROR: Connection refused to Redis"OpenSearch, ELK, CloudWatch
TracesCaminho de uma requisição entre serviçosAPI GW → ms-pix → ms-wallets → DB (total: 340ms)Jaeger, Dynatrace, Tempo

🔍 Analogia: Imagine investigar um acidente de carro. Métricas = velocímetro (mostra os números no momento). Logs = depoimento das testemunhas ("vi o carro frear às 14:32"). Traces = câmera de segurança (mostra o caminho completo do carro, quadro a quadro). Cada um dá uma perspectiva diferente — juntos, você reconstrói o que aconteceu.

Como se complementam:

  • Métrica diz que algo está errado (error rate subiu pra 5%)
  • Log diz o que está errado ("timeout connecting to Redis on port 6379")
  • Trace diz onde está errado (ms-payments → Redis call = 30s timeout no span 4)

💡 No PicPay... o fluxo típico de investigação é: alerta de métrica dispara → SRE abre o dashboard Grafana → identifica o serviço afetado → consulta logs no OpenSearch → abre o trace no Jaeger pra ver exatamente qual chamada entre serviços está lenta. Tudo correlacionado pelo trace_id.

3 Métricas e Prometheus

Prometheus é o sistema de métricas mais usado em Kubernetes. Ele "raspa" (scrape) endpoints /metrics dos seus serviços e armazena as séries temporais.

Tipos de métrica:

TipoO que fazExemplo
CounterSó sobe (conta eventos acumulados)http_requests_total = 1.523.847
GaugeSobe e desce (valor atual)cpu_usage_percent = 72.3
HistogramDistribui valores em bucketsrequest_duration_seconds (p50, p95, p99)
SummarySimilar ao histogram, calcula percentisresponse_time{quantile="0.99"} = 0.210

PromQL básico (linguagem de query do Prometheus):

# Taxa de requests por segundo nos últimos 5 minutos
rate(http_requests_total[5m])

# Percentil 99 de latência
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

# Taxa de erro (% de 5xx)
rate(http_requests_total{status=~"5.."}[5m])
/ rate(http_requests_total[5m]) * 100

# Uso de CPU dos pods do ms-pix
container_cpu_usage_seconds_total{namespace="pix"}

🌡️ Analogia: Prometheus é o termômetro digital da casa. A cada 15 segundos ele mede a temperatura (scrape). Guarda o histórico pra você ver a evolução. PromQL é a pergunta: "qual foi a temperatura máxima na última hora?" ou "qual cômodo está mais quente?"

💡 No PicPay... usamos VictoriaMetrics como backend de longa retenção (Prometheus guarda por padrão só 15 dias). As métricas são raspadas pelo OTel Collector e enviadas via prometheusremotewrite pro VictoriaMetrics. Dashboards Grafana consultam ambos.

4 Logs Estruturados

Log estruturado é log em formato JSON — cada campo tem nome e valor, o que permite buscas, filtros e agregações automáticas. O oposto é log em texto livre (difícil de parsear).

AspectoTexto LivreJSON Estruturado
ExemploERROR 2026-03-25 Payment failed for user 123{"level":"ERROR","timestamp":"2026-03-25T14:30:00Z","msg":"Payment failed","user_id":"123"}
BuscaRegex, grep (frágil)Filtro por campo: user_id=123 AND level=ERROR
DashboardDifícil de agregarFácil: "top 10 erros por tipo"

Campos obrigatórios num bom log:

  • timestamp — Quando aconteceu (ISO 8601)
  • level — Severidade: DEBUG, INFO, WARN, ERROR, FATAL
  • message — O que aconteceu (texto descritivo)
  • service — Qual serviço gerou o log
  • trace_id — Correlação com traces (conecta os 3 pilares)
  • user_id / request_id — Contexto da operação
{
  "timestamp": "2026-03-25T14:30:02.341Z",
  "level": "ERROR",
  "service": "ms-pix",
  "trace_id": "abc123def456",
  "message": "Timeout connecting to Redis",
  "host": "ms-pix-7b8d9-xkf2p",
  "duration_ms": 30000,
  "error_type": "ConnectionTimeoutException"
}

📝 Analogia: Log em texto livre é uma carta escrita à mão — legível pra humanos, difícil pra computadores. Log em JSON é um formulário preenchido — cada campo no lugar certo, fácil de filtrar e organizar automaticamente.

💡 No PicPay... logs vão pra OpenSearch (fork do Elasticsearch). O pipeline é: app → stdout (JSON) → Filebeat/Fluent Bit (coleta) → Kafka (buffer) → Flink/Logstash (processamento) → OpenSearch (armazenamento e busca). Sempre em JSON estruturado com trace_id pra correlação.

5 Distributed Tracing

Quando uma requisição passa por vários microserviços, como saber onde ela ficou lenta? Distributed tracing rastreia o caminho completo da requisição, serviço por serviço.

Conceitos-chave:

  • Trace — A jornada completa de uma requisição (do início ao fim)
  • Span — Uma etapa individual do trace (ex: chamada ao banco, request HTTP)
  • Trace ID — Identificador único que conecta todos os spans de uma requisição
  • Parent/Child — Spans formam uma árvore (o span pai chama o span filho)
  • Context Propagation — O trace ID é passado nos headers HTTP entre serviços

Exemplo visual de um trace:

Trace ID: abc-123-def-456    Total: 340ms

├─ [API Gateway]              0ms ─── 340ms  (340ms)
│  ├─ [ms-pix]                5ms ─── 330ms  (325ms)
│  │  ├─ [ms-wallets]        10ms ─── 120ms  (110ms)
│  │  │  └─ [PostgreSQL]     15ms ───  80ms  ( 65ms) ← OK
│  │  ├─ [ms-fraud]         125ms ─── 180ms  ( 55ms) ← OK
│  │  └─ [Redis cache]      185ms ─── 315ms  (130ms) ← LENTO!
│  └─ [Response]            320ms ─── 330ms  ( 10ms)

📦 Analogia: Trace é como rastrear uma encomenda dos Correios. O Trace ID é o código de rastreamento. Cada span é um ponto de passagem ("saiu do centro de distribuição SP", "chegou em BH", "saiu pra entrega"). Se a encomenda atrasou, você vê exatamente em qual etapa travou.

💡 No PicPay... usamos Jaeger e Dynatrace pra visualizar traces. O trace_id é propagado automaticamente via headers (traceparent no padrão W3C). Quando um Pix demora mais que o esperado, o SRE abre o trace e vê exatamente qual microserviço ou banco está lento — sem precisar adivinhar.

6 Dashboards e Grafana

Grafana é a ferramenta de dashboards mais usada no ecossistema cloud. Conecta em dezenas de fontes de dados (Prometheus, VictoriaMetrics, OpenSearch, CloudWatch) e exibe tudo num único lugar.

Anatomia de um bom dashboard:

  • Os 4 Golden Signals no topo — Latência, Tráfego, Erros, Saturação (visão rápida)
  • Gráficos de tendência — Linha do tempo mostrando evolução (últimas 1h, 6h, 24h)
  • Tabelas de detalhamento — Top endpoints mais lentos, top erros por tipo
  • Variáveis de filtro — Dropdown pra selecionar serviço, namespace, ambiente
  • Links rápidos — Logs no OpenSearch, traces no Jaeger, runbook no Confluence

Os 4 Golden Signals (recap):

SinalMétrica PromQLO que indica
Latênciahistogram_quantile(0.99, rate(http_duration_bucket[5m]))Experiência do usuário
Tráfegorate(http_requests_total[5m])Volume de carga
Errosrate(http_requests_total{status=~"5.."}[5m])Taxa de falha
Saturaçãocontainer_memory_usage_bytes / container_spec_memory_limitRisco de esgotamento

🎛️ Analogia: Grafana é o painel do carro. Velocímetro (latência), conta-giros (tráfego), luz de alerta (erros), marcador de combustível (saturação). Tudo no campo de visão do motorista, sem precisar abrir o capô pra saber como o motor está.

💡 No PicPay... todo microserviço tem um dashboard Grafana padrão gerado automaticamente. O SRE de plantão (on-call) usa Grafana como primeira tela de investigação. Temos dashboards por serviço, por squad, por domínio (Pix, carteira, cartão) e dashboards de infra (Kafka, Redis, RDS). Mais de 500 dashboards ativos.

7 Alertas

Alerta é uma notificação automática que dispara quando uma métrica ultrapassa um threshold. O objetivo: detectar problemas antes que o cliente perceba.

Bom alerta vs Mau alerta:

CaracterísticaBom AlertaMau Alerta
AçãoRequer ação humana imediataInformativo (ninguém faz nada)
PrecisãoDispara quando tem problema realFalso positivo constante
ContextoDiz o que está errado e link pro runbook"Alerta: CPU alta" (sem contexto)
ThresholdBaseado em SLO e impacto ao usuárioNúmero arbitrário ("CPU > 80%")

Alert Fatigue: Quando alertas disparam demais (falsos positivos), o time para de prestar atenção — e quando vem um alerta real, ninguém responde. É o "menino que gritava lobo" da tecnologia.

Severidades:

  • P1 — Crítico: Impacto ao cliente, precisa agir AGORA (ex: Pix fora do ar)
  • P2 — Alto: Degradação significativa, agir em até 30 min
  • P3 — Médio: Problema isolado, pode esperar horário comercial
  • P4 — Baixo: Informativo, sem impacto direto
# Exemplo de alerta no Prometheus (AlertManager)
groups:
  - name: ms-pix-alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5..",service="ms-pix"}[5m])
              / rate(http_requests_total{service="ms-pix"}[5m]) > 0.05
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "ms-pix error rate acima de 5%"
          runbook: "https://confluence.picpay.com/runbook/ms-pix-errors"

🚨 Analogia: Alerta bom é o alarme de incêndio — só toca quando tem fogo de verdade, e todo mundo sabe o que fazer (evacuar). Alerta ruim é o alarme do carro do vizinho — toca toda noite, ninguém liga mais, e quando roubam o carro de verdade ninguém reage.

💡 No PicPay... alertas P1/P2 notificam via PagerDuty (telefone/SMS do on-call). P3/P4 vão pro Slack. Todo alerta tem link pro runbook. O time de O11y revisa alertas periodicamente pra eliminar falsos positivos e reduzir alert fatigue. Mais de 100 alertas configurados.

8 SLI/SLO na Prática

Vimos SLI/SLO no módulo de SRE. Agora vamos ver como implementar de verdade — escolhendo métricas e definindo alvos realistas.

Escolhendo bons SLIs:

  • Disponibilidade: % de requests com sucesso (status != 5xx)
  • Latência: % de requests abaixo de um threshold (ex: < 300ms)
  • Corretude: % de respostas com dado correto (mais difícil de medir)
  • Throughput: Volume de operações completadas por segundo

Exemplo prático — Pix PicPay:

SLISLOMedição
Disponibilidade99.95% das transações com sucesso1 - (count 5xx / count total) por janela de 30 dias
Latência p9999% dos Pix concluídos em < 2shistogram_quantile(0.99, rate(pix_duration_bucket[30d]))
Latência p5050% dos Pix concluídos em < 500mshistogram_quantile(0.50, rate(pix_duration_bucket[30d]))

Não exagere nos "nines":

  • 99.9% = 43 min de downtime/mês — bom pra maioria dos serviços
  • 99.99% = 4 min de downtime/mês — caro demais pra serviços não-críticos
  • Dica: Comece com SLO mais relaxado e aperte conforme ganha confiança

🎯 Analogia: SLO é como meta de corrida. Se você nunca correu, não define "vou correr uma maratona em 3 horas". Começa com "vou completar 5km em menos de 40 minutos" — e vai apertando conforme treina. SLO agressivo demais engessa o time; relaxado demais não protege o cliente.

💡 No PicPay... SLOs são definidos por domínio (Pix, Carteira, Pagamentos) e revisados trimestralmente. O dashboard de error budget mostra quanto "saldo" cada serviço ainda tem no mês. Se um serviço queima 80% do budget na primeira semana, o time congela releases e prioriza estabilidade.

9 OpenTelemetry (OTel)

OpenTelemetry é o padrão open-source pra instrumentação de métricas, logs e traces. Em vez de usar SDKs diferentes pra cada ferramenta (Prometheus, Jaeger, Datadog), você instrumenta uma vez com OTel e envia pra qualquer backend.

Componentes do OTel:

  • SDK — Biblioteca que você adiciona no código (Java, Go, Python, etc.)
  • Auto-instrumentação — Captura métricas/traces automaticamente (sem mudar código)
  • OTel Collector — Agente que recebe, processa e exporta telemetria
  • OTLP — OpenTelemetry Protocol — protocolo padrão de envio de dados

Fluxo típico:

┌──────────┐    OTLP     ┌──────────────┐    export    ┌──────────────┐
│  App     │───────────▶│  OTel        │────────────▶│  Backends    │
│  + SDK   │  (gRPC/    │  Collector   │             │              │
│          │   HTTP)    │  (recebe,    │  ┌──────────▶ Prometheus   │
└──────────┘            │   filtra,    │  │          │  VictoriaM.  │
                        │   enriquece) │──┤          │  Jaeger      │
                        └──────────────┘  │          │  OpenSearch  │
                                          └──────────▶ Dynatrace   │
                                                     └──────────────┘

Instrumentação automática vs manual:

TipoComo funcionaQuando usar
AutomáticaJava agent / Go middleware — captura HTTP, DB, gRPC sem mudar código80% dos casos (início rápido)
ManualVocê cria spans e métricas customizadas no códigoLógica de negócio específica (ex: "tempo do cálculo de IOF")

🔌 Analogia: OTel é um adaptador universal de tomada. Em vez de ter um carregador diferente pra cada país (Prometheus, Jaeger, Datadog), você usa um adaptador universal (OTel) que funciona em qualquer lugar. Instrumenta uma vez, envia pra qualquer backend.

💡 No PicPay... o OTel Collector roda como DaemonSet/Deployment em todos os clusters EKS. As libs lib-java-olly (Java) e lib-go-instrumentation (Go) adicionam instrumentação automática. O Collector recebe via OTLP, processa (filtra, enriquece, faz batch) e envia pra VictoriaMetrics (métricas), Jaeger/Dynatrace (traces) e OpenSearch (logs).

10 Troubleshooting com Observabilidade

Quando um problema acontece em produção, o método reverso é o mais eficiente: comece pela ponta do usuário e vá voltando até encontrar a causa raiz.

Método Reverso (UI → Backend → Infra):

1. 🖥️  UI/Cliente
   → O que o usuário está vendo? Erro? Lentidão? Timeout?

2. 📊 Dashboards (Grafana)
   → Qual serviço está com error rate alto ou latência degradada?

3. 📋 Logs (OpenSearch)
   → Qual o erro exato? Timeout? NullPointer? Connection refused?

4. 🔗 Traces (Jaeger)
   → Qual span está lento? Qual dependência está falhando?

5. ⚙️  Infra (kubectl, CloudWatch)
   → Pod reiniciando? OOMKilled? CPU saturada? Disco cheio?

6. 🗄️  Dependências (DB, Redis, Kafka)
   → Slow queries? Redis sem memória? Kafka lag alto?

Dicas práticas:

  • Correlacione sempre — Use o trace_id pra ligar métrica + log + trace
  • "O que mudou?" — 80% dos problemas vêm de deploy recente (git log + ArgoCD history)
  • Compare com baseline — "Está lento comparado a quê?" Use o mesmo horário de ontem/semana passada como referência
  • Não assuma, verifique — "Deve ser o banco" não é diagnóstico. Abra o dashboard do RDS e comprove
  • Documente — Achou a causa? Atualize o runbook pra próxima vez ser mais rápido

🕵️ Analogia: Troubleshooting é trabalho de detetive. Você chega na cena do crime (alerta disparou), coleta evidências (métricas, logs, traces), cruza informações (correlação), elimina suspeitos (descarta hipóteses), e prende o culpado (identifica a causa raiz). O pior erro é prender o inocente (fix errado que não resolve o problema).

💡 No PicPay... o fluxo de incidente segue esse método. O on-call recebe o alerta, abre Grafana (métricas), consulta OpenSearch (logs), abre Jaeger (traces), e verifica infra com kubectl. Se não resolver em 15 minutos, escala pro tech lead. Todo incidente P1/P2 tem postmortem obrigatório com timeline, causa raiz e ações preventivas.
