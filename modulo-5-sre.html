Conceitos SRE

Confiabilidade como prática de engenharia

1 O que é SRE?

SRE (Site Reliability Engineering) é uma disciplina criada pelo Google para garantir que sistemas em produção sejam confiáveis, escaláveis e eficientes. O lema é: "Hope is not a strategy" — esperança não é estratégia.

Em vez de "torcer pra não cair", o SRE define métricas, automatiza processos e trata confiabilidade como um problema de engenharia, não de sorte.

🏥 Analogia: Se o sistema fosse um hospital, o Dev seria o cirurgião (constrói a solução) e o SRE seria o médico de UTI — monitora sinais vitais 24/7, tem protocolos de emergência prontos e previne que o paciente piore.

Pilares do SRE:

  • Automação sobre trabalho manual — Se você faz algo mais de 2x, automatize
  • SLIs/SLOs/SLAs — Medir confiabilidade com números, não com "achismo"
  • Error budget — Quanto de falha é aceitável antes de parar de lançar features
  • Postmortems blameless — Quando algo falha, aprender sem culpar pessoas
  • Redução de toil — Eliminar trabalho repetitivo e manual

💡 No PicPay... o time de SRE monitora centenas de microserviços em produção. Quando um serviço degrada, alertas disparam automaticamente (via Prometheus + Grafana + PagerDuty), runbooks guiam a resposta, e postmortems documentam lições aprendidas.

2 SRE vs DevOps

SRE e DevOps compartilham a mesma filosofia (colaboração Dev + Ops), mas têm abordagens diferentes. O Google define: "SRE é uma implementação específica de DevOps".

AspectoDevOpsSRE
FocoCultura e colaboraçãoEngenharia e métricas
Abordagem"Quebrar silos entre Dev e Ops""Tratar ops como problema de software"
MétricasDORA (deploy freq, lead time)SLI/SLO/SLA, error budget
AutomaçãoCI/CD, IaCEliminar toil, self-healing
IncidentesProcesso de respostaRunbooks, postmortems, on-call
Filosofia"Todos são responsáveis""50% ops, 50% engenharia"

🔧 Analogia: DevOps é o movimento (tipo "todo mundo deveria cuidar da saúde"). SRE é a prescrição médica específica (dieta X, exercício Y, exame Z a cada 6 meses).

💡 No PicPay... praticamos os dois. DevOps como cultura (squads donas do ciclo de vida) e SRE como prática (SLOs definidos, error budget monitorado, on-call com rotação semanal, postmortems obrigatórios pra incidentes P1/P2).

3 SLI, SLO e SLA

Três siglas que formam a base de como o SRE mede confiabilidade. Sem isso, você não sabe se o sistema está "bom" ou "ruim".

SiglaSignificadoExemplo (Pix PicPay)
SLIService Level Indicator — a métrica que você mede% de transações Pix concluídas em < 2s
SLOService Level Objective — o alvo interno99.9% das transações Pix em < 2s
SLAService Level Agreement — contrato com o cliente (com penalidade)"Se Pix ficar fora > 4h/mês, reembolsamos taxas"

📏 Analogia: SLI = o termômetro (mede a febre). SLO = a meta ("manter abaixo de 37°C"). SLA = o contrato com o plano de saúde ("se passar de 39°C e não tratar, reembolso").

Relação entre eles:

  • SLI é o dado bruto (latência p99, taxa de erro, disponibilidade)
  • SLO é a meta interna — mais rigorosa que o SLA (margem de segurança)
  • SLA é o compromisso externo — se violar, tem penalidade financeira/contratual

💡 No PicPay... o SLO interno do Pix é 99.95% de disponibilidade (mais rigoroso que o BACEN exige). Isso dá margem: se o BACEN exige 99.9%, nós miramos 99.95% pra nunca chegar perto do limite contratual. Os SLIs são monitorados em dashboards Grafana em tempo real.

4 Error Budget

Error budget é quanto de indisponibilidade você pode "gastar" por mês sem violar o SLO. É um conceito revolucionário: falhar um pouco é OK — faz parte do orçamento.

SLOError Budget/mêsDowntime permitido
99.9%0.1%~43 minutos
99.95%0.05%~22 minutos
99.99%0.01%~4 minutos

💰 Analogia: Pense no error budget como mesada. Você tem R$43 (minutos) por mês pra "gastar" em falhas. Se gastar tudo na primeira semana com um deploy ruim, o resto do mês é congelamento de releases — sem deploys novos até o mês virar.

Como funciona na prática:

  • Budget sobrando → Pode lançar features mais rápido, fazer experimentos
  • Budget acabando → Freezar releases, focar em estabilidade e correções
  • Budget estourado → Postmortem obrigatório, plano de ação, só fixes de confiabilidade

💡 No PicPay... acompanhamos o error budget dos serviços críticos (Pix, pagamentos, carteira) semanalmente. Se o Pix consome 60% do budget em uma semana por causa de um incidente, o restante do mês prioriza estabilidade sobre features novas.

5 Toil — Trabalho Manual Repetitivo

Toil é qualquer trabalho manual, repetitivo, automátizable, que não gera valor duradouro. O Google define a meta: SREs devem gastar no máximo 50% do tempo com toil. Os outros 50% são engenharia (automação, melhorias, ferramentas).

🧹 Analogia: Toil é varrer o chão todo dia. Engenharia é comprar um robô aspirador. Se você só varre, nunca sobra tempo pra melhorar nada. A meta é automatizar a varrição e usar o tempo livre pra coisas que importam.

Exemplos de toil:

  • Reiniciar pods manualmente quando travam → automatizar com liveness probes
  • Rodar scripts de limpeza toda segunda → agendar com CronJob
  • Responder tickets repetitivos ("escala o serviço X") → criar self-service ou HPA
  • Rotacionar certificados na mão → usar cert-manager automático
  • Verificar logs manualmente → criar alertas no Prometheus/Grafana

Características do toil:

  • Manual — Requer intervenção humana
  • Repetitivo — Acontece várias vezes
  • Automátizavel — Uma máquina poderia fazer
  • Sem valor duradouro — Não melhora o sistema permanentemente
  • Escala linearmente — Cresce proporcionalmente ao tamanho do sistema

💡 No PicPay... o time de SRE automatizou rotação de secrets, scaling de clusters, e resposta a alertas comuns. GMUDs repetitivas (ex: escalar réplicas) ganharam self-service via PR no repo de infra — o dev abre PR, pipeline aplica, zero toil pro SRE.

6 Os 4 Sinais Dourados (Golden Signals)

O Google definiu que 4 métricas são suficientes pra saber se um serviço está saudável. Se você só pudesse monitorar 4 coisas, seriam estas:

SinalO que medeExemplo
LatênciaTempo de resposta das requisiçõesp99 = 200ms, p50 = 45ms
TráfegoVolume de requisições1.500 req/s no pico
ErrosTaxa de requisições com falha0.3% retornando HTTP 5xx
SaturaçãoQuão "cheio" o recurso estáCPU 78%, memória 85%

🚗 Analogia: Se o serviço fosse um carro: Latência = velocidade do carro. Tráfego = quantos carros na estrada. Erros = carros que quebraram no meio do caminho. Saturação = combustível restante no tanque.

Variação — método RED e USE:

  • RED (pra serviços) — Rate (taxa), Errors (erros), Duration (duração)
  • USE (pra infra) — Utilization (utilização), Saturation (saturação), Errors (erros)

💡 No PicPay... todo microserviço tem um dashboard Grafana com os 4 Golden Signals. O template padrão inclui: request rate (req/s), error rate (% 5xx), latência (p50/p95/p99) e saturação (CPU/memória dos pods). Alertas disparam quando qualquer sinal sai do range esperado.

7 Health Checks — Liveness e Readiness

No Kubernetes, probes são verificações automáticas que o cluster faz pra saber se um container está saudável. São a base pra self-healing.

ProbePerguntaSe falhar...
Liveness"Você está vivo?"Kubernetes reinicia o container
Readiness"Você está pronto pra receber tráfego?"Kubernetes para de enviar requests
Startup"Você já terminou de inicializar?"Kubernetes espera antes de checar liveness

🏪 Analogia: Liveness = "a loja existe?" (se queimou, reconstrói). Readiness = "a loja está aberta?" (se está em reforma, não manda clientes). Startup = "a loja está inaugurando?" (espera terminar a obra antes de verificar se está aberta).

Configuração típica em YAML:

livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 3

Cuidado: Liveness probe muito agressiva (timeout curto) pode causar restart loop — o container reinicia antes de terminar de subir.

💡 No PicPay... todo microserviço expõe /health/live e /health/ready. O readiness verifica se o serviço consegue conectar no banco e no Redis. Se o banco cair, o pod fica "not ready" e para de receber tráfego — sem derrubar o pod inteiro.

8 Runbooks — Guias de Resposta a Incidentes

Runbook é um documento passo-a-passo que diz exatamente o que fazer quando um alerta dispara. O objetivo: qualquer pessoa do on-call consegue resolver, mesmo às 3h da manhã.

📋 Analogia: Runbook é o manual de instruções do micro-ondas. Quando o alerta dispara ("comida não esquenta"), você abre o manual e segue os passos: "1. Verifique se está na tomada. 2. Verifique o fusível. 3. Se nada funcionar, chame técnico."

Estrutura de um bom runbook:

## Alerta: ms-payments — High Error Rate (> 5%)

### Diagnóstico
1. Verificar dashboard: grafana.picpay.com/d/ms-payments
2. Checar logs: kubectl logs -n payments -l app=ms-payments --tail=100
3. Verificar dependências: banco (RDS), cache (Redis), fila (Kafka)

### Causas Comuns
- Banco de dados lento → verificar slow queries no RDS
- Redis timeout → verificar conexões no ElastiCache
- Deploy recente → verificar último deploy no ArgoCD

### Mitigação
1. Se deploy recente → rollback: argocd app rollback ms-payments
2. Se banco lento → escalar read replicas
3. Se Redis → reiniciar pool de conexões

### Escalação
- Não resolveu em 15min → chamar tech lead do time
- P1 confirmado → abrir war room no Slack #incident-response

💡 No PicPay... cada alerta no PagerDuty tem um link pra runbook no Confluence. Quando o on-call recebe o alerta às 3h, o runbook diz exatamente o que checar, em que ordem, e quando escalar. Postmortems atualizam os runbooks quando descobrem cenários novos.

9 Microserviços — Arquitetura Distribuída

Em vez de um único sistema gigante (monolito), dividimos em serviços pequenos e independentes, cada um responsável por uma função específica.

AspectoMonolitoMicroserviços
DeployTudo junto — 1 artefatoCada serviço independente
EscalaEscala tudo (mesmo o que não precisa)Escala só o que precisa
FalhaBug pode derrubar tudoFalha isolada (um serviço cai, resto funciona)
TecnologiaUma stack pra tudoCada serviço pode usar stack diferente
ComplexidadeCódigo complexo, infra simplesCódigo simples, infra complexa

🏭 Analogia: Monolito = uma fábrica gigante que faz tudo (parafuso até carro completo). Se o setor de pintura quebra, a fábrica inteira para. Microserviços = várias fábricas pequenas especializadas. Se a de pintura quebra, as outras continuam — e você pode ter 2 fábricas de pintura pra redundância.

Diagrama simplificado:

         ┌─────────────┐
         │   API GW    │ ← Kong / ALB
         │  (Gateway)  │
         └──────┬──────┘
       ┌────────┼────────┐
       ▼        ▼        ▼
  ┌─────────┐ ┌──────┐ ┌─────────┐
  │ms-users │ │ms-pix│ │ms-wallet│
  │  (Go)   │ │(Java)│ │ (Kotlin)│
  └────┬────┘ └──┬───┘ └────┬────┘
       │         │           │
       ▼         ▼           ▼
   [Postgres] [Postgres]  [Redis+DB]

💡 No PicPay... rodamos centenas de microserviços em Kubernetes (EKS). Cada squad é dona dos seus serviços (you build it, you run it). Comunicação entre serviços é via REST (síncrono) e Kafka (assíncrono). O Kong faz roteamento de API Gateway na borda.

10 Terraform — Infrastructure as Code (IaC)

Terraform permite definir infraestrutura como código — em vez de clicar no console AWS pra criar recursos, você escreve arquivos .tf e o Terraform cria tudo automaticamente.

🏗️ Analogia: Se criar infra no console é construir uma casa tijolo por tijolo (manual), Terraform é ter a planta da casa em arquivo — você manda "construir" e a casa sai exatamente igual toda vez. Precisa mudar algo? Altera a planta e reconstrói.

Ciclo de vida Terraform:

# 1. Inicializar (baixa providers)
terraform init

# 2. Planejar (mostra o que vai mudar — NÃO aplica)
terraform plan

# 3. Aplicar (cria/altera recursos de verdade)
terraform apply

# 4. Destruir (remove tudo que foi criado)
terraform destroy

Exemplo — criar um bucket S3:

resource "aws_s3_bucket" "logs" {
  bucket = "picpay-logs-prod"
  
  tags = {
    Environment = "production"
    Team        = "sre"
  }
}

resource "aws_s3_bucket_versioning" "logs" {
  bucket = aws_s3_bucket.logs.id
  versioning_configuration {
    status = "Enabled"
  }
}

Conceitos-chave:

  • State — Arquivo que guarda o estado atual da infra (terraform.tfstate). Guardado remotamente (S3 + DynamoDB)
  • Provider — Plugin que fala com a cloud (aws, gcp, azurerm, kubernetes)
  • Module — Bloco reutilizável de código Terraform (tipo função)
  • Plan — Preview das mudanças. Sempre rodar antes de apply!
  • Terragrunt — Wrapper que gerencia múltiplos ambientes (dev, staging, prod)

💡 No PicPay... toda infraestrutura é gerenciada por Terraform + Terragrunt em repositórios *-infra. Ninguém cria recursos clicando no console AWS. O ciclo é: dev abre PR no repo de infra → pipeline roda terraform plan → revisor aprova → merge roda terraform apply. State é armazenado em S3 com lock via DynamoDB.
