Glossário Técnico

Referência completa de termos técnicos para o Command Center N1 — pesquisável, com filtros por categoria e explicações práticas

71 termos

Cluster

Infra

Conjunto de servidores (nodes) que trabalham juntos como uma unidade. Se um node cai, os outros assumem a carga — garantindo alta disponibilidade.

💡 No PicPay... rodamos clusters EKS (Kubernetes gerenciado pela AWS) com dezenas de nodes distribuídos em múltiplas AZs.

Node

Infra

Servidor individual dentro de um cluster. Pode ser uma máquina física ou virtual (EC2). Cada node roda containers/pods e tem CPU, memória e disco próprios.

💡 No PicPay... nodes são instâncias EC2 gerenciadas pelo EKS. O auto-scaler adiciona ou remove nodes conforme a demanda.

Load Balancer

Infra

Componente que distribui tráfego entre múltiplos servidores. Evita que um único servidor fique sobrecarregado. Faz health check pra remover servidores doentes.

💡 No PicPay... usamos ALBs (Application Load Balancer) na frente dos serviços, roteando por hostname e path.

Auto Scaling

Infra

Mecanismo que aumenta ou diminui automaticamente a quantidade de servidores/pods conforme a demanda. Escala pra cima no pico e pra baixo na madrugada.

💡 No PicPay... o HPA (Horizontal Pod Autoscaler) escala pods por CPU/memória, e o Cluster Autoscaler adiciona nodes EC2 quando necessário.

CDN

Infra

Content Delivery Network — rede de servidores espalhados pelo mundo que cacheia conteúdo estático (imagens, CSS, JS) perto do usuário final, reduzindo latência.

💡 No PicPay... usamos CloudFront (CDN da AWS) pra servir assets estáticos do app e site com latência mínima.

WAF

Infra

Web Application Firewall — protege aplicações web filtrando e monitorando tráfego HTTP. Bloqueia ataques como SQL Injection, XSS e DDoS na borda.

💡 No PicPay... o WAF fica na frente do CloudFront e ALB, bloqueando requests maliciosos antes de chegarem nos microsserviços.

API Gateway

Infra

Ponto de entrada único para todas as APIs. Cuida de autenticação, rate limiting, roteamento e transformação de requests antes de encaminhar ao serviço correto.

💡 No PicPay... usamos o Kong como API Gateway — ele roteia requests do app mobile pra os microsserviços internos, aplicando auth e rate limit.

VPC

Infra

Virtual Private Cloud — rede privada isolada na nuvem. Dentro dela ficam subnets, servidores, bancos de dados. Você controla quem entra e quem sai.

💡 No PicPay... cada ambiente (dev, staging, prod) tem sua própria VPC isolada, com subnets públicas (ALBs) e privadas (serviços, bancos).

Subnet

Infra

Sub-rede dentro de uma VPC. Pode ser pública (com acesso à internet via Internet Gateway) ou privada (sem acesso direto — usa NAT Gateway pra sair).

💡 No PicPay... ALBs ficam em subnets públicas, microsserviços e bancos em subnets privadas — ninguém acessa direto da internet.

Security Group

Infra

Firewall virtual que controla tráfego de entrada e saída de recursos AWS. Define quais portas e IPs podem acessar uma instância EC2, RDS, etc.

💡 No PicPay... cada serviço tem SGs restritos — RDS só aceita conexão do cluster EKS, nunca da internet diretamente.

EC2

AWS

Elastic Compute Cloud — máquina virtual na AWS. Você escolhe CPU, memória e disco, liga e usa. Paga por hora/segundo de uso. Base de toda computação na nuvem.

💡 No PicPay... os nodes do EKS rodam em EC2s. A maioria dos devs não acessa EC2 direto — o Kubernetes gerencia tudo por cima.

S3

AWS

Simple Storage Service — armazenamento de objetos com durabilidade de 11 noves (99.999999999%). Guarda arquivos, logs, backups, imagens em buckets.

💡 No PicPay... S3 armazena logs (via Flink), backups de RDS, artifacts de CI/CD e dados históricos pro Databricks.

RDS

AWS

Relational Database Service — banco de dados relacional gerenciado. AWS cuida de backup, patches, replicação. Suporta PostgreSQL, MySQL, Oracle, SQL Server.

💡 No PicPay... usamos RDS PostgreSQL com Multi-AZ — failover automático se a AZ primária cair.

EBS

AWS

Elastic Block Store — disco virtual persistente anexado a instâncias EC2. Os dados sobrevivem mesmo se o EC2 for desligado. Tipos: gp3 (uso geral), io2 (alta performance).

💡 No PicPay... volumes EBS são usados pelos nodes EKS e bancos RDS. Snapshots periódicos garantem backup.

IAM

AWS

Identity and Access Management — controla quem pode acessar o quê na AWS. Users, Groups, Roles e Policies definem permissões granulares.

💡 No PicPay... cada microsserviço tem uma IAM Role com permissões mínimas (least privilege). Devs usam assume-role temporário.

Lambda

AWS

Serverless — você sobe código (função) e a AWS executa sob demanda. Sem servidor pra gerenciar. Paga apenas pelo tempo de execução (ms). Escala automaticamente.

💡 No PicPay... Lambdas são usadas pra tarefas event-driven — processar eventos S3, webhooks, rotinas leves que não justificam um microsserviço.

CloudWatch

AWS

Serviço de monitoramento nativo da AWS. Coleta métricas (CPU, rede, disco), armazena logs e dispara alarmes quando thresholds são atingidos.

💡 No PicPay... CloudWatch é backup pra alertas de componentes AWS puros (RDS, ALB, S3). O principal é Prometheus + Grafana.

Route 53

AWS

Serviço DNS da AWS. Traduz nomes de domínio (api.picpay.com) em endereços IP. Suporta health checks e roteamento geográfico/weighted.

💡 No PicPay... Route 53 gerencia todos os domínios picpay.com e ppay.me, roteando tráfego pra ALBs e CloudFront.

EKS

AWS

Elastic Kubernetes Service — Kubernetes gerenciado pela AWS. Você não precisa instalar/manter o control plane. AWS cuida da disponibilidade do master.

💡 No PicPay... EKS é o coração da infra. Todos os microsserviços rodam em clusters EKS com dezenas de nodes em múltiplas AZs.

ECR

AWS

Elastic Container Registry — registry privado de imagens Docker na AWS. Armazena e versiona as imagens que o EKS puxa pra rodar os containers.

💡 No PicPay... toda imagem Docker buildada no CI/CD é pushada pro ECR antes do deploy no EKS.

Pod

Kubernetes

Menor unidade deployável no Kubernetes. Um pod roda 1 ou mais containers que compartilham rede e storage. Na prática, 1 pod = 1 instância do seu serviço.

💡 No PicPay... cada microsserviço roda como pods — o HPA escala a quantidade de pods conforme CPU/memória.

Deployment

Kubernetes

Recurso que gerencia pods. Define quantas réplicas manter, qual imagem usar e como fazer rollout. Se um pod morrer, o Deployment recria automaticamente.

💡 No PicPay... todo microsserviço tem um Deployment (ou Rollout do Argo) que garante N réplicas rodando sempre.

Service

Kubernetes

Abstração de rede que expõe pods. Como pods morrem e nascem com IPs diferentes, o Service dá um IP/DNS fixo pra acessar o conjunto de pods.

💡 No PicPay... cada microsserviço tem um Service (ClusterIP) que outros serviços usam pra se comunicar internamente.

Namespace

Kubernetes

Divisão lógica dentro de um cluster. Isola recursos (pods, services, configmaps) entre times ou ambientes. Como pastas organizacionais.

💡 No PicPay... cada microsserviço geralmente tem seu próprio namespace (ex: ms-wallets, ms-piggy).

ConfigMap

Kubernetes

Objeto que armazena configurações não-sensíveis (URLs, feature flags, limites) como pares chave-valor. Injetado nos pods como variáveis de ambiente ou arquivos.

💡 No PicPay... ConfigMaps guardam configs de ambiente (URLs de serviços, timeouts). Alterações requerem restart dos pods.

Secret

Kubernetes

Similar ao ConfigMap, mas pra dados sensíveis (senhas, tokens, chaves). Armazenado em base64 (não é criptografia!). Usar com ExternalSecrets pra segurança real.

💡 No PicPay... Secrets são sincronizados do AWS Secrets Manager via ExternalSecrets — nunca hardcoded no repo.

Ingress

Kubernetes

Recurso que gerencia acesso externo aos Services do cluster. Define regras de roteamento HTTP (por hostname, path) e terminação TLS/SSL.

💡 No PicPay... Ingress conecta o ALB ao Service do microsserviço, roteando tráfego externo pra dentro do cluster.

HPA

Kubernetes

Horizontal Pod Autoscaler — escala automaticamente o número de pods baseado em métricas (CPU, memória, custom metrics). Define min e max de réplicas.

💡 No PicPay... a maioria dos serviços usa HPA com target de 70% CPU. Alguns usam KEDA pra escalar por Kafka lag ou SQS.

PVC

Kubernetes

PersistentVolumeClaim — requisição de storage persistente. O pod pede disco, o cluster provisiona (geralmente EBS). Dados sobrevivem ao restart do pod.

💡 No PicPay... PVCs são usados por bancos (PostgreSQL em pods), Prometheus e outros serviços stateful.

DaemonSet

Kubernetes

Garante que um pod rode em TODOS os nodes do cluster. Usado pra agentes de coleta de logs, métricas ou segurança que precisam estar em toda máquina.

💡 No PicPay... Filebeat (coleta de logs) e Dynatrace OneAgent rodam como DaemonSets — presentes em todo node.

StatefulSet

Kubernetes

Como um Deployment, mas pra workloads que precisam de identidade persistente (nomes fixos, storage fixo). Usado pra bancos de dados e sistemas distribuídos.

💡 No PicPay... Kafka brokers e alguns bancos internos rodam como StatefulSets com PVCs dedicados.

CronJob

Kubernetes

Job que roda em schedule (cron). Cria pods temporários em horários definidos pra executar tarefas periódicas (limpeza, relatórios, conciliação).

💡 No PicPay... CronJobs fazem conciliação financeira diária, limpeza de dados temporários e geração de relatórios batch.

Liveness Probe

Kubernetes

Verificação periódica de saúde do container. Se falhar N vezes consecutivas, o Kubernetes mata e recria o container. Detecta pods "travados".

💡 No PicPay... todo serviço expõe /health/liveness — se travar (deadlock), o K8s recria o pod automaticamente.

Readiness Probe

Kubernetes

Verifica se o pod está pronto pra receber tráfego. Diferente do liveness — não mata o pod, apenas remove do load balancer até que fique saudável novamente.

💡 No PicPay... /health/readiness verifica conexão com banco e dependências. Pod só recebe tráfego quando tudo OK.

Helm

Kubernetes

Gerenciador de pacotes do Kubernetes. Empacota YAMLs complexos em "charts" reutilizáveis com valores configuráveis (values.yaml). Versionável e reproduzível.

💡 No PicPay... usamos o chart picpay-ms-v2 como base pra todos os microsserviços — padroniza deploy, probes, resources.

DNS

Rede

Domain Name System — "agenda telefônica da internet". Traduz nomes legíveis (api.picpay.com) em IPs (54.231.0.12). Sem DNS, teríamos que decorar números.

💡 No PicPay... Route 53 é o DNS autoritativo. Internamente, CoreDNS resolve nomes de Services K8s (svc.cluster.local).

TCP/UDP

Rede

Protocolos de transporte. TCP é confiável (garante entrega, ordem) — usado por HTTP, banco de dados. UDP é rápido mas sem garantia — usado por DNS, streaming, games.

💡 No PicPay... 99% do tráfego é TCP (APIs REST, gRPC, bancos). UDP aparece em DNS resolution e métricas (StatsD).

HTTP/HTTPS

Rede

Protocolo de comunicação web. HTTP transfere dados em texto aberto. HTTPS adiciona criptografia TLS — obrigatório pra dados sensíveis. Porta padrão: 80 (HTTP) e 443 (HTTPS).

💡 No PicPay... toda comunicação externa é HTTPS. Internamente entre microsserviços, pode ser HTTP (dentro da VPC privada).

Latência

Rede

Tempo que um pacote leva pra ir do ponto A ao ponto B. Medida em milissegundos (ms). Quanto menor, melhor a experiência do usuário. Afetada por distância física e saltos de rede.

💡 No PicPay... monitoramos latência P50, P95 e P99 de toda API. SLO típico: P99 < 500ms pra APIs críticas.

IP Address

Rede

Endereço numérico que identifica um dispositivo na rede. IPv4: 4 octetos (192.168.1.1). IPv6: endereço mais longo pra suportar mais dispositivos. Pode ser público ou privado.

💡 No PicPay... pods K8s recebem IPs privados (dentro da VPC). ALBs têm IPs públicos expostos via DNS.

Firewall

Rede

Barreira de segurança que filtra tráfego de rede. Pode bloquear portas, IPs, protocolos. Na AWS: Security Groups (stateful) e Network ACLs (stateless).

💡 No PicPay... temos camadas: CloudFlare (borda) → WAF (camada 7) → Security Groups (camada 4) → Network ACL (subnet).

SLI

Observabilidade

Service Level Indicator — métrica que mede o comportamento do serviço do ponto de vista do usuário. Exemplo: "% de requests com latência < 300ms".

💡 No PicPay... SLIs comuns: taxa de sucesso (2xx/total), latência P99, disponibilidade por janela de tempo.

SLO

Observabilidade

Service Level Objective — meta interna pro SLI. "99.9% dos requests devem completar em < 300ms". Se violar o SLO, é hora de priorizar confiabilidade.

💡 No PicPay... SLOs são definidos por serviço crítico. Violação do SLO consome error budget e pode bloquear deploys.

SLA

Observabilidade

Service Level Agreement — contrato formal (geralmente com cliente/parceiro). Mais rígido que SLO — violar SLA pode gerar multas, créditos ou penalidades contratuais.

💡 No PicPay... SLAs são compromissos com parceiros (bancos, BACEN). SLOs internos são mais agressivos que SLAs pra ter margem.

Error Budget

Observabilidade

Margem de erro permitida. Se SLO é 99.9%, o error budget é 0.1% (43 min/mês). Enquanto houver budget, pode deployar features. Se esgotar, foco em confiabilidade.

💡 No PicPay... error budget é monitorado no Grafana. Quando fica baixo, deploys são pausados até recuperar a margem.

Latência P50/P95/P99

Observabilidade

Percentis de latência. P50 = mediana (metade mais rápida). P95 = 95% dos requests foram mais rápidos que esse valor. P99 = captura quase todo mundo, incluindo outliers lentos.

💡 No PicPay... alertamos no P99 — se 1% dos usuários está com latência alta, já investigamos. P50 mostra a experiência "normal".

Métricas

Observabilidade

Dados numéricos coletados ao longo do tempo (time series). CPU, memória, request rate, error rate, latência. Armazenadas em bancos como Prometheus/VictoriaMetrics.

💡 No PicPay... métricas vão pro VictoriaMetrics via OTel Collector. Consultadas via PromQL no Grafana.

Logs

Observabilidade

Registros textuais de eventos do sistema. Cada linha = um evento (request recebido, erro, warning). Estruturados (JSON) são muito mais fáceis de buscar que texto livre.

💡 No PicPay... logs vão pro OpenSearch (via Filebeat/Kafka/Flink). Buscas no Grafana via datasource OpenSearch.

Traces

Observabilidade

Rastreamento distribuído de um request através de múltiplos serviços. Cada "span" mostra quanto tempo gastou em cada serviço. Essencial pra debugar latência.

💡 No PicPay... traces vão pro Dynatrace e Jaeger via OTel. Permitem ver o caminho completo: Kong → BFF → Core → DB.

Alerta

Observabilidade

Notificação automática quando uma métrica cruza um threshold. Severidades: P1 (crítico, acorda gente), P2 (alto, responde em minutos), P3-P5 (baixo, próximo business day).

💡 No PicPay... alertas disparam no Slack e PagerDuty. Command Center monitora 24/7 e aciona on-call quando necessário.

Dashboard

Observabilidade

Painel visual com gráficos de métricas em tempo real. Permite monitorar saúde do sistema num relance — CPU, error rate, latência, throughput por serviço.

💡 No PicPay... Grafana é a plataforma principal. Cada time tem dashboards com métricas RED (Rate, Errors, Duration).

Prometheus

Observabilidade

Sistema de monitoramento open-source. Coleta métricas via pull (scraping), armazena como time-series e permite queries com PromQL. Padrão de mercado.

💡 No PicPay... Prometheus coleta métricas dos pods via service discovery do K8s. Backend principal: VictoriaMetrics (compatível, mais escalável).

Grafana

Observabilidade

Plataforma de visualização de dados. Conecta a múltiplos datasources (Prometheus, OpenSearch, CloudWatch) e cria dashboards interativos com alertas integrados.

💡 No PicPay... Grafana é o "painel de controle" do Command Center. Organizado por orgs (PF, PJ, Tech Cross, CX).

MTTR

Observabilidade

Mean Time To Recovery — tempo médio pra restaurar o serviço após uma falha. Quanto menor, melhor. Inclui detecção + diagnóstico + correção + validação.

💡 No PicPay... meta de MTTR varia por severidade. P1: < 30 min. Runbooks e automações ajudam a reduzir.

MTTD

Observabilidade

Mean Time To Detect — tempo médio pra detectar que algo está errado. Depende de qualidade dos alertas, dashboards e monitoramento. Ideal: < 5 minutos.

💡 No PicPay... MTTD é reduzido com alertas bem calibrados. O Command Center monitora 24/7 e detecta anomalias rapidamente.

CI/CD

DevOps

Continuous Integration / Continuous Delivery. CI: código é integrado e testado automaticamente a cada commit. CD: código aprovado é deployado automaticamente em produção.

💡 No PicPay... pipeline: commit → build → testes → imagem Docker → push ECR → ArgoCD sincroniza no EKS. Tudo automático.

Pipeline

DevOps

Sequência automatizada de passos (build, test, deploy). Cada passo depende do anterior. Se um falha, o pipeline para e notifica. Executado por ferramentas como GitHub Actions, Jenkins.

💡 No PicPay... pipelines rodam no GitHub Actions (Tekton pra alguns). Incluem lint, testes unitários, scan de segurança e deploy.

GitOps

DevOps

Prática onde o Git é a fonte de verdade pra infra e deploys. Toda mudança vai pro repo (declarativa) e um operador (ArgoCD) sincroniza o estado real com o desejado.

💡 No PicPay... GitOps com ArgoCD: repos -infra têm o estado desejado, ArgoCD detecta drift e sincroniza automaticamente.

ArgoCD

DevOps

Ferramenta GitOps pra Kubernetes. Monitora repos Git e garante que o cluster reflita o estado declarado. Detecta drift, faz sync automático e rollback.

💡 No PicPay... ArgoCD é o motor de deploy. Toda mudança no repo -infra é sincronizada automaticamente no cluster EKS.

Terraform

DevOps

Ferramenta de Infrastructure as Code (IaC). Define infra em código declarativo (HCL). Executa plan (preview) e apply (cria/altera recursos). Versionável e auditável.

💡 No PicPay... Terraform gerencia recursos AWS (RDS, S3, VPC, IAM). Repos Alfred centralizam IaC por ambiente.

IaC

DevOps

Infrastructure as Code — gerenciar infra com código versionado em vez de configuração manual. Garante reprodutibilidade, auditoria e consistência entre ambientes.

💡 No PicPay... toda infra é IaC: Terraform pra AWS, Helm charts pra K8s, ArgoCD pra deploy. Zero configuração manual.

Blue/Green

DevOps

Estratégia de deploy com dois ambientes idênticos. Blue = produção atual. Green = nova versão. Quando green está pronto, troca o tráfego. Rollback instantâneo: volta pro blue.

💡 No PicPay... blue/green é usado pra mudanças de infra críticas. Pra microsserviços, preferimos canary (mais granular).

Canary

DevOps

Deploy gradual: nova versão recebe 5% do tráfego → 20% → 50% → 100%. Se métricas degradarem em qualquer step, rollback automático. Menor risco que blue/green.

💡 No PicPay... Argo Rollouts faz canary deploy com análise Prometheus. Se error rate subir, rollback automático sem intervenção humana.

Rollback

DevOps

Reverter pra versão anterior quando um deploy causa problemas. Pode ser automático (canary detectou degradação) ou manual (operador decide). Velocidade é crítica.

💡 No PicPay... rollback via ArgoCD: reverter o commit no repo -infra ou usar argo rollouts undo. Em segundos a versão anterior volta.

Feature Flag

DevOps

Toggle que liga/desliga funcionalidades sem deploy. Permite deploy com feature desativada, testar com % de usuários, e rollback instantâneo (desligar a flag).

💡 No PicPay... Houston é o sistema de feature flags. Permite rollout gradual por % de usuários, segmentação e kill switch.

Incident Management

Processos

Processo estruturado pra lidar com incidentes. Detecção → Triagem → Comunicação → Mitigação → Resolução → Postmortem. Objetivo: restaurar o serviço o mais rápido possível.

💡 No PicPay... o Command Center lidera o processo. Incidentes P1/P2 abrem war room, comunicação via Slack e StatusPage.

Postmortem

Processos

Documento escrito após um incidente significativo. Sem culpa (blameless). Documenta: o que aconteceu, timeline, impacto, causa raiz e ações preventivas.

💡 No PicPay... todo incidente P1/P2 tem postmortem obrigatório em até 5 dias. Publicado no Confluence e revisado em reunião.

Runbook

Processos

Guia passo a passo pra resolver problemas conhecidos. "Se alerta X disparar, faça Y". Reduz MTTR porque o operador não precisa pensar do zero — segue o procedimento.

💡 No PicPay... runbooks ficam no Confluence linkados aos alertas do Grafana. O Command Center consulta antes de escalar.

Escalação

Processos

Processo de encaminhar um incidente pra alguém com mais contexto ou autoridade. L1 (CC) tenta resolver → L2 (SRE) → L3 (time dono do serviço). Cada nível tem mais expertise.

💡 No PicPay... CC N1 resolve o que consegue e escala pro time responsável via PagerDuty quando necessário.

On-Call

Processos

Escala de plantão — pessoa responsável por responder incidentes fora do horário comercial. Rotação semanal típica. Ferramenta de acionamento: PagerDuty, OpsGenie.

💡 No PicPay... times críticos têm on-call 24/7 via PagerDuty. O Command Center é o primeiro ponto de contato.

RCA

Processos

Root Cause Analysis — análise de causa raiz. Técnica pra descobrir POR QUÊ o incidente aconteceu (não só O QUÊ). Método dos "5 porquês" é o mais comum.

💡 No PicPay... toda RCA faz parte do postmortem. Sem RCA sólida, as ações corretivas podem resolver só o sintoma, não a causa.

DRP

Processos

Disaster Recovery Plan — plano pra recuperar sistemas após desastre (região AWS cair, corrupção de dados). Define RTO (tempo máximo pra voltar) e RPO (quanto de dado pode perder).

💡 No PicPay... DRP inclui failover pra outra região AWS, restore de backups S3/RDS e procedimentos de comunicação.

RTO

Processos

Recovery Time Objective — tempo máximo aceitável pra restaurar o serviço após desastre. Ex: RTO de 4h significa que em 4 horas o sistema precisa estar de pé.

💡 No PicPay... RTO varia por criticidade. Serviços financeiros core: RTO < 1h. Serviços internos: RTO < 4h.

RPO

Processos

Recovery Point Objective — quantidade máxima de dados que pode ser perdida. Ex: RPO de 1h = backup a cada hora. RPO de 0 = replicação síncrona (zero data loss).

💡 No PicPay... bancos críticos (saldo, transações) têm RPO ~0 com replicação Multi-AZ. Logs têm RPO de ~15 min.

Change Management

Processos

Processo formal pra aprovar mudanças em produção. Avalia risco, impacto, plano de execução e rollback. Garante que mudanças sejam controladas e auditáveis.

💡 No PicPay... mudanças em produção exigem GMUD (Gestão de Mudanças) aprovada no Jira GSTI antes da execução.

Kafka

Dados

Plataforma de streaming distribuído. Produtores publicam mensagens em tópicos, consumidores leem. Alta vazão, persistente, ordenado por partição. Backbone de eventos assíncronos.

💡 No PicPay... Kafka é o sistema nervoso — eventos de transação, notificações, logs e integração entre serviços passam por tópicos Kafka.

SQS

Dados

Simple Queue Service — fila de mensagens da AWS. Desacopla serviços: produtor envia mensagem, consumidor processa quando puder. Garante at-least-once delivery.

💡 No PicPay... SQS é usado pra tarefas assíncronas que não precisam da complexidade do Kafka (ex: envio de emails, processamento de arquivos).

Redis

Dados

Banco de dados in-memory (chave-valor). Ultra rápido (~1ms). Usado como cache, session store, rate limiter, distributed lock. Dados voláteis (podem ser perdidos).

💡 No PicPay... Redis cacheia saldos, sessões, feature flags e é usado como distributed lock pra operações críticas.

DynamoDB

Dados

Banco NoSQL serverless da AWS. Chave-valor e documentos. Escala automaticamente, latência single-digit ms. Paga por request (on-demand) ou capacidade provisionada.

💡 No PicPay... DynamoDB é usado por serviços que precisam de escala massiva com schema flexível (ex: metadados de notificação).

PostgreSQL

Dados

Banco de dados relacional open-source. Robusto, extensível, suporta JSON, full-text search e transações ACID. O banco relacional mais usado em startups e fintechs.

💡 No PicPay... PostgreSQL (via RDS) é o banco padrão pra microsserviços. Multi-AZ com failover automático.

MongoDB

Dados

Banco NoSQL orientado a documentos (JSON/BSON). Schema flexível, escala horizontal. Bom pra dados semi-estruturados que mudam frequentemente.

💡 No PicPay... MongoDB Atlas é usado por serviços que precisam de schema flexível (ex: cofrinhos, catálogos, atendimento CX).

ElastiCache

Dados

Serviço gerenciado de cache in-memory da AWS. Suporta Redis e Memcached. AWS cuida de patches, backup, failover. Você só usa como Redis gerenciado.

💡 No PicPay... ElastiCache Redis é o principal cache — clusters dedicados por domínio (wallets, auth, statement).

Replicação

Dados

Copiar dados automaticamente entre servidores. Síncrona (espera confirmação — zero data loss) ou assíncrona (mais rápida, risco mínimo de perda). Essencial pra HA e DR.

💡 No PicPay... RDS Multi-AZ usa replicação síncrona (zero data loss). Read replicas usam assíncrona (lag de ms).

Sharding

Dados

Dividir dados horizontalmente entre múltiplos bancos. Cada shard guarda um subset (ex: usuários A-M no shard 1, N-Z no shard 2). Escala capacidade total.

💡 No PicPay... Kafka usa partições (sharding nativo). Bancos SQL evitam sharding quando possível (complexidade alta).

Connection Pool

Dados

Pool de conexões pré-abertas com o banco de dados. Reusar conexões existentes é muito mais rápido que abrir uma nova a cada request. Tamanho do pool = tuning crítico.

💡 No PicPay... HikariCP é o connection pool padrão nos serviços Java. Pool size é ajustado por serviço conforme carga.
