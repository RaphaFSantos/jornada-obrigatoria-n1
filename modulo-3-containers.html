Containers — Docker e Kubernetes

Semana 3 — Isolamento, portabilidade e orquestração

1 O Problema que Containers Resolvem

Você cria uma aplicação em seu MacBook. Funciona perfeitamente. Aí você leva pra um servidor Linux em produção e... quebra! Por quê?

🎬 Analogia: Você filma um filme no seu estúdio (MacBook). Tudo perfeito. Aí você tenta exibir em outro cinema (servidor Linux) e o projetor é incompatível, a tela é pequena, o som não bate.

Diferenças que causam problema:

  • Sistema operacional diferente (macOS vs Linux)
  • Versão das bibliotecas diferentes (Python 3.8 vs 3.11)
  • Dependências ausentes
  • Variáveis de ambiente
  • Permissões de arquivo

A solução clássica (ruim): "Funciona na minha máquina" — dev fica ajustando servidores manualmente. Cada deploy é diferente. Pesadelo de manutenção.

A solução moderna (boa): Containers — você empacota a aplicação COM seu ambiente (SO, bibliotecas, tudo) em um arquivo. Esse arquivo roda em qualquer lugar — MacBook, Linux, Windows — idêntico.

💡 No PicPay... todo microserviço roda em um container Docker. Quando você faz deploy, está enviando o container inteiro — não quer dizer de dependências no servidor, sem "funciona em dev mas não em prod".

2 Docker — O Padrão de Containers

Docker é a ferramenta que cria e executa containers. Você escreve um Dockerfile (receita) e Docker empacota tudo em uma imagem.

📦 Analogia: Docker é como um navio porta-contêiner. O container é a caixa. A imagem Docker é o molde da caixa. Você faz um molde uma vez, depois produz milhares de caixas idênticas.

Conceitos principais:

  • Dockerfile — Arquivo de texto com instruções (base OS, instalar dependências, copiar código)
  • Imagem — Arquivo compilado (tipo .exe) — você faz docker build e gera uma imagem
  • Container — Instância rodando de uma imagem (tipo processo em execução)
  • Registry — Repositório de imagens (Docker Hub, ECR, GitHub Container Registry). Você faz docker push pra enviar a imagem lá

Exemplo de Dockerfile simples:

FROM python:3.11
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]

Cada linha é uma "camada". Docker cacheia essas camadas — se você só mudou seu código (.py), Docker não reinstala Python (mais rápido).

Executar um container:

docker run -p 8000:8000 -e DATABASE_URL=postgres://... minha-app:latest

💡 No PicPay... todo microserviço tem um Dockerfile. Na CI/CD, quando você faz push no GitHub, o pipeline automático faz docker build e faz docker push pra ECR (Elastic Container Registry da AWS). Depois Kubernetes puxa a imagem de lá e roda o container.

3 O Problema dos Containers (Scaling)

Containers resolvem isolamento e portabilidade, mas e quando você tem 100 containers rodando ao mesmo tempo? Quem gerencia?

  • Qual máquina roda qual container?
  • Se um container morre, quem relança?
  • Se tráfego sobe 10x, como escalo containers automaticamente?
  • Como faço load balancing entre containers?
  • Como faço rolling update sem downtime?

Solução anterior (ruim): Scripts bash. Pesadelo. Escalabilidade = morte.

Solução moderna (boa): Kubernetes — orquestrador de containers. Ele cuida de tudo.

💡 No PicPay... rodamos Kubernetes (EKS — Elastic Kubernetes Service da AWS). Você descreve "quero 3 replicas de ms-users rodando" e Kubernetes cuida de lançar, monitorar, escalar, fazer updates.

4 Kubernetes — Orquestrador de Containers

Kubernetes (K8s, "K" + 8 letras + "s") é um sistema que gerencia centenas ou milhares de containers.

🎭 Analogia: Se Docker é um ator (container), Kubernetes é o diretor do filme — dizendo qual ator entra em cena, quando sai, se um ator fica doente (container morre) quem substitui.

Conceitos principais:

  • Pod — Menor unidade. É como um container, mas tecnicamente é um wrapper. 99% dos casos = 1 container por pod
  • Deployment — Gerencia múltiplos pods da mesma aplicação. Você diz "quero 3 replicas de ms-users" — deployment cuida de criar 3 pods
  • Service — Expõe os pods pra rede (tipo load balancer). Clientes conectam no Service, que distribui pra pods
  • Node — Máquina (EC2) que roda pods. Cluster K8s = vários nodes
  • Namespace — Isolamento lógico (tipo carpetas). dev, staging, prod ficam em namespaces separados

Arquivo declarativo (YAML):

apiVersion: apps/v1
kind: Deployment
metadata:
  name: ms-users
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: app
        image: ecr.aws.com/ms-users:latest
        ports:
        - containerPort: 8000

Você escreve isso em YAML, faz kubectl apply -f deployment.yaml e Kubernetes cria 3 pods da imagem ms-users.

Recursos automáticos do K8s:

  • Self-healing — Pod morreu? Kubernetes relança
  • Rolling updates — Você faz deploy de v2. K8s desliga pods de v1 lentamente, liga v2. Zero downtime
  • Scaling automático (HPA) — Se CPU > 70%, adiciona mais pods
  • Load balancing — Service distribui tráfego entre pods

💡 No PicPay... cada microserviço tem um Deployment Kubernetes. Você escreve o YAML uma vez, faz push e pronto. Deploy é automático (Argo CD cuida disso). Se pod trava, Kubernetes relança. Se tráfego sobe, HPA escala pods automaticamente.

5 Anatomia de um Cluster Kubernetes

Um cluster K8s tem dois tipos de máquinas: Control Plane (governo) e Nodes (trabalhadores).

Control PlaneNodes
FunçãoGovernador — decide o que fazerTrabalhador — executa
ComponentesAPI Server, etcd, Scheduler, Controller ManagerKubelet, kube-proxy, container runtime (Docker)
QuantidadeNormalmente 3 (alta disponibilidade)1-1000+
AcessoProtegido (não roda aplicações user)Roda containers de apps

Fluxo de um deployment:

1. Você: kubectl apply -f deployment.yaml
2. API Server: recebe e valida
3. Scheduler: "qual node tem recurso? coloca pod no node-1"
4. Control Plane: "ei node-1, roda o container ms-users:latest"
5. Node-1: Kubelet puxa imagem do ECR e roda container
6. Pod está vivo!

💡 No PicPay... usamos EKS (Elastic Kubernetes Service) — AWS gerencia o Control Plane pra você. Você só paga pelos nodes (máquinas EC2) que rodam seus containers. Muito mais fácil que instalar Kubernetes do zero.

6 Container Registry — Repositório de Imagens

Imagens Docker precisam ficar armazenadas em um lugar — um registry. É como GitHub pra código, mas pra imagens Docker.

RegistryCaracterística
Docker Hub (público)Imagens públicas de tudo — redis, postgres, ubuntu
ECR (AWS)Private — onde colocamos nossas imagens PicPay
GitHub Container RegistryIntegrado ao GitHub, grátis pra públicas
ArtifactoryCorporativo — também guarda Maven, npm, etc

Workflow típico:

1. Você faz código local
2. Commit no GitHub
3. CI/CD (GitHub Actions):
   - docker build -t ms-users:v1.2.3 .
   - docker push 123456.dkr.ecr.sa-east-1.amazonaws.com/ms-users:v1.2.3
4. Kubernetes puxa imagem do ECR
5. Container roda

💡 No PicPay... imagens vão pro ECR. Cada build gera uma imagem com tag (commit SHA ou version). ECR é privado — apenas máquinas Kubernetes do PicPay podem puxar. Imagens antigas são deletadas automaticamente (lifecycles policy) pra economizar storage.

7 Containers Stateless vs Stateful

Stateless = o container não guarda estado (dados locais). Pode ser deletado e recreado sem problema.

Stateful = o container guarda estado (banco de dados, cache). Perder o container = perder dados.

StatelessStateful
ExemploAPI REST (ms-users, ms-wallets)PostgreSQL, Redis, Elasticsearch
EscalaçãoFácil — crie 5 cópiasComplexo — precisa sincronizar estado
DeployRolling update — sem problemaCuidado com downtime
StorageNão precisaVolume persistente (SSD EC2 ou EBS)
OrquestraçãoKubernetes (simples)Kubernetes + StatefulSet (complexo)

APIs REST são stateless — cada requisição é independente. Bancos de dados são stateful — dados precisam persistir. Kubernetes é bom pra stateless. Stateful em K8s é complicado (por isso muitos usam serviços gerenciados — RDS, ElastiCache).

💡 No PicPay... microserviços (ms-users, ms-wallets, ms-payments) são stateless — rodam em Deployments K8s. Bancos de dados (PostgreSQL, Redis) são gerenciados fora do K8s — usamos RDS (gerenciado AWS) e ElastiCache (gerenciado AWS). Muito mais simples e confiável.

8 Volumes — Armazenamento Persistente

Container normal: dados morrem quando ele morre. Se você precisa dados persistentes, precisa montar um volume.

Tipos de volume no K8s:

  • emptyDir — Pasta temporária enquanto pod roda. Quando pod morre, dados somem
  • hostPath — Pasta na máquina (node). Se pod muda de node, perde dados
  • Persistent Volume (PV) — Storage separado (EBS, EFS, GlusterFS). Persiste mesmo que pod morra
  • Persistent Volume Claim (PVC) — Pod solicita storage. K8s aloca um PV
  • Cloud storage — EBS (AWS), GCP Persistent Disk, Azure Disk. Mais confiável

Na prática: APIs stateless não precisam de volumes. Bancos de dados precisam — mas melhor usar serviços gerenciados (RDS) do que cuidar de storage no K8s.

💡 No PicPay... microserviços são stateless, não usam volumes. RDS e Redis são gerenciados (fora do K8s), então não temos que lidar com storage persistente no cluster.

9 Helm — Package Manager do Kubernetes

YAML puro é verboso. Se você tem 100 microserviços, cada um com 5 YAMLs... é caos. Helm empacota tudo em charts.

📦 Analogia: Kubernetes é como instalar software manualmente (configure tudo, copie cada arquivo). Helm é como um instalador (clique "next" e pronto).

Um Helm chart tem:

  • Chart.yaml — Metadados (nome, versão)
  • values.yaml — Variáveis (quantas replicas, qual imagem, limitações de resource)
  • templates/ — YAMLs Kubernetes em template (com variáveis)

Uso:

helm install ms-users ./ms-users-chart/ \
  --set replicas=3 \
  --set image.tag=v1.2.3

Isso cria 3 replicas da imagem v1.2.3 em um só comando. Sem Helm você digitaria 50+ linhas de YAML.

💡 No PicPay... cada microserviço tem um Helm chart. Argo CD usa os charts pra fazer deploy (mais pra frente, módulo 5). Você muda values.yaml (replicas, image tag) e Argo CD atualiza Kubernetes automaticamente.

10 Networking em Containers

Containers precisam se comunicar. Kubernetes oferece networking nativo.

Tipos de comunicação:

  • Pod-to-Pod — Pods no mesmo cluster conversam direto pelo IP (CNI — Container Network Interface cuida)
  • Pod-to-Service — Pods usam service name (ms-users:8000) em vez de IP direto. Service descobre IPs automaticamente
  • Ingress — Tráfego externo (internet) chega via Ingress (tipo reverse proxy)

Exemplo de comunicação:

Pod ms-wallets roda: http://ms-users:8000/api/users/123
(não precisa saber o IP exato do pod ms-users)
Service ms-users:
  - distribui pra 3 pods
  - faz load balancing
  - faz DNS resolution (ms-users → IP dinâmico)

DNS interno do K8s (Kubernetes DNS) resolve service names automaticamente. Mágica.

💡 No PicPay... microserviços conversam pelo service name dentro do cluster (ms-users, ms-wallets, etc — configurado em DNS do K8s). Tráfego externo entra via ALB (AWS) + Kong (API Gateway). Kong roteia pra serviços internos.
