Bancos de Dados

Semana 4 — Onde os dados realmente vivem: SQL, NoSQL e escolhas arquiteturais

1 Bancos de Dados Relacionais (SQL)

Um banco relacional organiza dados em tabelas. Imagine uma planilha Excel — linhas são registros, colunas são atributos, e há regras que garantem integridade.

📋 Analogia: Um banco relacional é como um arquivo de fichas. Cada ficha é um registro (linha), cada informação na ficha é um campo (coluna). As tabelas são gavetas diferentes organizadas por tipo de informação.

Características principais:

  • ACID — Atomicidade, Consistência, Isolamento, Durabilidade. Garantem que os dados nunca ficam "pela metade".
  • Schema rígido — Você define antecipadamente quais colunas existem, qual tipo cada uma tem, se é obrigatória.
  • Relacionamentos — Tabelas conectadas por chaves (Foreign Keys). Uma ordem está conectada a um cliente.
  • SQL — Linguagem padronizada pra consultar dados.

Bancos SQL mais comuns:

  • PostgreSQL — Open-source, poderoso, excelente pra produção. É o padrão no PicPay.
  • MySQL / MariaDB — Popular, simples, usado em muitos places.
  • Oracle — Corporativo, robusto, caro. Raro em startups.
  • SQL Server — Feito pela Microsoft, muito usado em enterprise.

Exemplo prático — uma consulta SQL simples:

SELECT nome, email, saldo 
FROM usuarios 
WHERE criado_em > '2024-01-01' 
  AND status = 'ativo'
ORDER BY saldo DESC
LIMIT 10;

💡 No PicPay... usamos PostgreSQL como nosso banco principal. Dados de transações, usuários, carteiras — tudo fica no Postgres, replicado em múltiplas zonas pra alta disponibilidade. Quando você vê um alerta "database connection timeout", o backend tentou conectar no Postgres e não conseguiu.

2 NoSQL — Bancos de Dados Não-Relacionais

SQL é ótimo pra dados estruturados e inter-relacionados. Mas e se os dados são não-estruturados? E se precisamos escalar horizontalmente pra milhões de requisições?

📚 Analogia: SQL é uma biblioteca tradicional com fichas catalogadas. NoSQL é mais como uma montanha de caixas — você manda dados lá, e o sistema arruma. Menos organização formal, mais velocidade e flexibilidade.

Tipos principais de NoSQL:

  • Document (MongoDB) — Dados em formato JSON/BSON. Muito flexível — o schema muda conforme precisa.
  • Key-Value (Redis) — Chave → Valor simples. Ultra-rápido. Ideal pra cache e sessões.
  • Wide-Column (HBase, Cassandra) — Colunas agrupadas em famílias. Escala horizontalmente sem limite.
  • Search (Elasticsearch) — Otimizado pra buscas em texto completo. Ótimo pra logs.

Quando escolher NoSQL?

  • Dados não-estruturados ou que mudam de forma frequente.
  • Precisa de escalabilidade horizontal (milhões de docs, não tabelas gigantes).
  • A velocidade é mais importante que a consistência garantida (eventual consistency).
  • Menos relacionamentos entre dados — mais dados isolados.

Exemplo — inserção em MongoDB (NoSQL):

db.usuarios.insertOne({
  _id: ObjectId(...),
  nome: "João Silva",
  email: "joao@example.com",
  telefones: ["+55 11 98765-4321"],
  endereco: {
    rua: "Av. Paulista",
    numero: 1000,
    cidade: "São Paulo"
  },
  metadata: {
    criado_em: new Date(),
    tags: ["premium", "verified"]
  }
});

💡 No PicPay... usamos MongoDB pra dados não-estruturados (eventos, configurações, caches de ofertas) e Redis pra cache de alta velocidade. O Postgres fica com os dados críticos — transações, saldos, usuários. A escolha depende do caso de uso.

3 PostgreSQL — O Banco Relacional do PicPay

PostgreSQL é open-source, confiável e poderoso. É o banco que você vai usar 90% do tempo no Command Center.

Características principais:

  • ACID — Garantias de integridade total. Transações são seguras.
  • Escalabilidade — Suporta bilhões de linhas sem perder performance.
  • Tipos de dados avançados — Arrays, JSON, UUID, tipos customizados.
  • Índices — Buscas rápidas mesmo em tabelas enormes.
  • Replicação e backup — Recuperação de falhas é nativa.

Operações CRUD básicas (Create, Read, Update, Delete):

-- CREATE: Inserir
INSERT INTO usuarios (nome, email, saldo) 
VALUES ('João', 'joao@example.com', 1000.00);

-- READ: Consultar
SELECT * FROM usuarios WHERE id = 1;

-- UPDATE: Atualizar
UPDATE usuarios SET saldo = 1500.00 WHERE id = 1;

-- DELETE: Deletar
DELETE FROM usuarios WHERE id = 1;

Indíces — a arma secreta pra performance:

-- Sem índice: SELECT * FROM usuarios WHERE email = '...'
-- escaneia TODA a tabela (lento!)

-- Com índice:
CREATE INDEX idx_usuarios_email ON usuarios(email);
-- agora a busca é O(log n) — rápido!

Índices têm custo — ocupam espaço e deixam escritas mais lentas. Use com critério, especialmente em colunas que você consulta frequentemente.

💡 No PicPay... o PostgreSQL roda em Aurora (RDS gerenciado pela AWS) com replicação multi-AZ — se uma zona cair, há outra com cópia atualizada. Quando você faz um deploy de código novo, o schema precisa ser compatível com a versão anterior (blue-green) — sem migrations que quebrem o banco.

4 Redis — Cache e Dados em Memória

Redis é um banco de dados em memória. Dados vivem na RAM, não no disco. Resultado: velocidade de milissegundos pra operações que no Postgres levariam 100ms+.

⚡ Analogia: Se PostgreSQL é um arquivo gigante no arquivo, Redis é uma mochila com as coisas que você usa todo dia. Acesso rápido, mas capacidade limitada.

Casos de uso principais:

  • Cache — Guardar resultado de queries caras. "Qual o saldo do usuário 123? Já perguntei hoje, tá em cache!"
  • Sessões — Dados de usuário autenticado. Rápido, expira automaticamente.
  • Filas — Processa jobs assincronamente. "Enviar email" entra na fila, worker processa.
  • Rate Limiting — Contar requisições por IP. "Esse IP já fez 1000 requisições em 1 minuto, bloqueia!"
  • Contadores** — Views, likes, clicks. Incrementar é O(1) — instantâneo.

Exemplo — Cache em Redis:

-- Guardar dados em cache (com expiral automática)
SET usuario:123:saldo "1500.00" EX 3600
-- EX 3600 = expira em 3600 segundos (1 hora)

-- Recuperar
GET usuario:123:saldo
-- Retorna: "1500.00" (rápido!)

-- Se não achar, consulta no Postgres e guarda novamente

Problema: Redis mantém tudo em RAM. Se o servidor cair, tudo se perde. É por isso que Redis é cache, não a única fonte de verdade.

💡 No PicPay... usamos Redis pra cache de saldo (atualizado a cada transação), autenticação (sessões), e rate-limiting de APIs. O Redis é replicado (Sentinel ou Cluster) — se um nó cai, há backups prontos. Quando o Redis fica indisponível, as buscas vão direto pro Postgres (mais lento, mas funciona).

5 MongoDB — Documentos Flexíveis

MongoDB armazena documentos JSON. Não é tabelinha rigidez — é mais tipo "guarda esse JSON, não importa se outro JSON tem campos diferentes".

Quando escolher MongoDB?

  • Dados semiestruturados — logs, eventos, configurações.
  • O schema muda frequentemente — adicionar campos sem migration.
  • Escalabilidade horizontal — replicar dados em múltiplos servidores.
  • Menos relacionamentos entre dados — documentos são mais autônomos.

Exemplo — Query em MongoDB:

db.eventos.find({
  usuario_id: 123,
  tipo: "transacao",
  criado_em: { $gte: new Date("2024-01-01") }
}).sort({ criado_em: -1 }).limit(10);

Diferenças principais vs Postgres:

PostgreSQLMongoDB
EstruturaTabelas rígidas com schema definidoDocumentos JSON flexíveis
TransaçõesACID completoACID (mas menos garantido)
RelacionamentosForeign Keys, JOINs fáceisReferências, menos relações
EscalabilidadeVertical (máquina maior)Horizontal (mais máquinas)
Uso idealDados críticos, estruturadosDados flexíveis, logs, eventos

💡 No PicPay... usamos MongoDB pra cofrinhos, ofertas de crédito e eventos não-estruturados. O Postgres continua sendo a fonte de verdade pra transações e saldos — dados que precisam de garantias ACID.

6 Transactions e Consistency

Uma transação agrupa várias operações em uma unidade atômica. Ou todas são executadas, ou nenhuma é. Sem meio termo.

💰 Analogia: Transferência bancária: debita da conta A, credita na conta B. Se falhar no meio, A perdeu dinheiro mas B não recebeu? Isso não pode acontecer. Transação garante: ambas as operações acontecem, ou nenhuma acontece.

Exemplo — transferência de dinheiro:

BEGIN TRANSACTION;
  UPDATE contas SET saldo = saldo - 100 WHERE id = 1;
  UPDATE contas SET saldo = saldo + 100 WHERE id = 2;
COMMIT;
-- Se der erro no meio, tudo reverte (ROLLBACK).

Níveis de isolamento (do mais fraco ao mais forte):

  • READ UNCOMMITTED — Lê dados que ainda não foram "commitados". Perigoso!
  • READ COMMITTED — Lê só dados confirmados. Padrão da maioria dos bancos.
  • REPEATABLE READ — Mesma linha, lida duas vezes, tem o mesmo valor.
  • SERIALIZABLE — Transações correm sequencialmente, sem concorrência. Ultra-seguro, muito lento.

No PicPay, a maioria das operações usa READ COMMITTED ou REPEATABLE READ — balanço entre segurança e performance.

💡 No PicPay... transações são críticas em operações financeiras. Um débito sem o crédito correspondente é um bugs que pode custar muito dinheiro. O Postgres garante ACID — mesmo com falhas, transações incompletas são revertidas automaticamente.

7 Migrations — Alterando Schema em Produção

Código muda. Bancos de dados precisam acompanhar. Uma migration é um script que muda o schema (cria tabela, adiciona coluna, cria índice, etc).

Problema clássico: você muda o código e o schema em produção, mas a versão antiga do código ainda está rodando. Ele tenta escrever na coluna que não existe mais — crash!

Solução — Migrations que são backwards-compatible:

-- ❌ PERIGO: Delete coluna
ALTER TABLE usuarios DROP COLUMN email_antigo;
-- Código antigo quebra!

-- ✅ SEGURO: Adiciona coluna, código novo usa, depois remove
ALTER TABLE usuarios ADD COLUMN email_novo VARCHAR(255);
-- Deploy novo código (que usa email_novo)
-- Depois, remover a coluna antiga
ALTER TABLE usuarios DROP COLUMN email_antigo;

Ferramentas comuns pra migrations:

  • Flyway — Java/Spring, bem simples, SQL scripts.
  • Liquibase — XML ou YAML, mais flexível.
  • Alembic — Python, integrado com SQLAlchemy.
  • Knex.js — Node.js, programática.

💡 No PicPay... as migrations são parte do deploy. Todo merge na main dispara uma migration automática. Se a migration falhar, o deploy inteiro cancela — nunca deixa código novo com schema velho.

8 N+1 Queries — A Armadilha Comum

N+1 é quando você faz 1 query e depois N queries adicionais em loop. Resultado: lentidão terrível.

Exemplo do problema:

-- Query 1: Busca 100 usuários
usuarios = SELECT * FROM usuarios LIMIT 100;

-- Loop (N=100 queries!):
for usuario in usuarios:
  endereco = SELECT * FROM enderecos WHERE usuario_id = usuario.id;
  
-- Total: 1 + 100 = 101 queries! 🐌

Solução — JOIN:

-- 1 query, traz tudo:
SELECT u.*, e.* 
FROM usuarios u
LEFT JOIN enderecos e ON e.usuario_id = u.id
LIMIT 100;
-- Total: 1 query! ⚡

Regra de ouro: sempre que você escreve um loop que consulta o banco, pare e pense: "Isso pode ser uma query só?"

💡 No PicPay... N+1 é uma das causas mais comuns de performance degradada. Quando um endpoint fica lento (>1s), a primeira coisa a verificar é se tem um loop consultando o banco. Ferramentas como NewRelic e Dynatrace mostram isso — procure por "DB calls".
