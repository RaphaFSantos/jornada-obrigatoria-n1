AWS para SRE

Semana 2 — Conhecendo a nuvem: EC2, VPC, RDS, Load Balancers

1 O que é Cloud Computing?

Historicamente, você tinha que comprar e manter seus próprios servidores — máquinas físicas caríssimas que ocupam espaço, geram calor, precisam de refrigeração e eletricidade. Cloud computing muda isso: você aluga computação sob demanda.

🏠 Analogia: Antes você comprava sua casa (servidor físico). Cloud é como alugar um apartamento — você paga mensalmente, o dono cuida da manutenção, e você não se preocupa com nada.

Modelos de nuvem:

  • IaaS (Infrastructure as a Service) — Você aluga máquinas, redes, storage. Exemplo: AWS EC2
  • PaaS (Platform as a Service) — Você aluga uma plataforma pronta (banco de dados, CI/CD). Exemplo: Heroku, AWS RDS
  • SaaS (Software as a Service) — Você acessa o software pela internet (ninguém se preocupa com infra). Exemplo: Slack, Jira, Salesforce

💡 No PicPay... usamos IaaS (AWS) pra rodar toda a infraestrutura — máquinas virtuais (EC2), bancos de dados (RDS), armazenamento (S3), redes (VPC). A nuvem permite que a gente scale de forma elástica — quando a Black Friday vem com 10x tráfego, podemos adicionar máquinas automaticamente.

2 Regiões e Availability Zones (AZs)

AWS tem data centers espalhados pelo mundo em regiões. Cada região tem múltiplas Availability Zones — prédios diferentes no mesmo estado/país, separados fisicamente.

🌎 Analogia: Região = estado (São Paulo). Availability Zone = cidade dentro do estado (São Paulo, Campinas, São José dos Campos). Se um prédio em SP queimar, a AZ em Campinas continua rodando.

Regiões AWS principais:

  • us-east-1 — Leste dos EUA (Virgínia) — a mais barata e antiga
  • us-west-2 — Oeste dos EUA (Oregon)
  • eu-west-1 — Europa (Irlanda)
  • sa-east-1 — São Paulo (Brasil) — onde roda PicPay
  • ap-northeast-1 — Tóquio (Japão)

Para reduzir latência, você coloca infraestrutura perto dos usuários. Como PicPay atende Brasil, rodamos em sa-east-1.

💡 No PicPay... rodamos em sa-east-1 (São Paulo). Distribuímos aplicações em múltiplas AZs dentro da mesma região pra tolerância a falhas — se um prédio cair, o outro continua rodando. Isso é multi-AZ deployment.

3 EC2 — Elastic Compute Cloud

EC2 é máquina virtual na nuvem — você liga, acessa, instala o que quiser (Docker, Node.js, Go). Você paga pelo tempo que a máquina fica ligada.

💻 Analogia: Se servidor físico é comprar um computador, EC2 é alugar um computador que você usa quando precisa e desliga quando termina.

Conceitos-chave:

  • Instance type — Tamanho (t2.micro = fraco, m5.xlarge = potente). Você escolhe CPU, memória e velocidade de rede
  • AMI (Amazon Machine Image) — Imagem (tipo Docker image) — vem com SO e software pré-instalado
  • Key pair — Chave SSH pra conectar na máquina remotamente
  • Security group — Firewall — controla quais portas são acessíveis
  • Auto Scaling Group — Cria ou deleta máquinas automaticamente conforme a carga

Exemplo de comando pra conectar:

ssh -i my-key.pem ec2-user@ec2-54-231-0-12.sa-east-1.compute.amazonaws.com

💡 No PicPay... os microserviços rodam em Kubernetes (EKS), que por baixo roda em EC2s. A maioria dos devs não acessa EC2 direto — o Kubernetes cuida da orquestração.

4 VPC — Virtual Private Cloud

VPC é sua rede privada na AWS. Dentro dela, você cria subnets (sub-redes), coloca máquinas, bancos de dados e controla quem acessa o quê.

🏢 Analogia: Você tem um prédio (VPC). Dentro dele há vários andares (subnets). Cada andar tem uma função — um é pra web, outro pra banco de dados. Você controla quem entra e sai.

Componentes de uma VPC:

  • Subnet — Sub-rede. Pode ser pública (tem acesso à internet) ou privada
  • Internet Gateway — Porta de entrada pra internet
  • NAT Gateway — Permite que máquinas privadas iniciarem conexões pro exterior
  • Route table — Define pra onde o tráfego vai (routing)
  • Network ACL / Security Group — Firewall (ACL é mais rígido, Security Group é mais flexível)

💡 No PicPay... temos VPCs isoladas por ambiente (dev, staging, prod). Dentro da VPC prod, máquinas rodando APIs são colocadas em subnets privadas e acessadas pelo ALB (que fica na subnet pública). Ninguém acessa direto — tudo passa pelo load balancer.

5 RDS — Relational Database Service

RDS é banco de dados gerenciado — você não precisa instalar PostgreSQL ou MySQL em um EC2. AWS cuida de backup, patches, replicação. Você só usa.

📦 Analogia: Antes (manual): você compra um baú, coloca coisas lá, tranca à noite, faz backups manualmente. RDS: você aluga um baú, AWS cuida de trancas, backups, e você só pede pra abrir/fechar.

Bancos de dados suportados:

  • PostgreSQL — Open source, poderoso, é o preferido
  • MySQL — Popular, mais rápido, menos features
  • MariaDB — Fork do MySQL
  • Oracle — Corporativo, caro
  • SQL Server — Microsoft

Características importantes:

  • Multi-AZ — Replica pra outra AZ automaticamente (failover automático)
  • Read replicas — Cópias de leitura em outras regiões (reduz latência)
  • Backup automático — Retenção de 7 a 35 dias
  • Encryption — Dados criptografados em repouso e em trânsito

💡 No PicPay... usamos RDS PostgreSQL com Multi-AZ habilitado. Se a AZ primária cai, o PostgreSQL falha automaticamente pra uma réplica em outra AZ — zero downtime. Backups são armazenados no S3 e retidos por 35 dias.

6 S3 — Simple Storage Service

S3 é armazenamento de objetos — você sobe arquivos (docs, imagens, logs, backups) e eles ficam ali, acessíveis pela internet ou internamente.

📁 Analogia: Se disco rígido é seu armário em casa, S3 é um deposito alugado do outro lado da cidade. Você guarda qualquer coisa lá, e a AWS garante que não vai perder.

Características:

  • Durabilidade 99.999999999% (11 noves) — Praticamente impossível perder dados
  • Buckets — Contêineres. Você cria um bucket e sobe arquivos lá
  • Objetos — Os arquivos dentro do bucket. Cada um tem uma key (caminho)
  • Lifecycle policies — Mover pra armazenamento mais barato conforme a idade (Glacier pra backups antigos)
  • Versioning — Manter histórico de todas as versões de um arquivo
  • Criptografia — SSE-S3 (padrão) ou SSE-KMS (mais controlado)

Preço varia: $0.023 por GB pra dados "quentes" (acessados frequentemente). Cai dramaticamente pra Glacier ($0.004/GB) — dados acessados raramente.

💡 No PicPay... usamos S3 pra: logs (Kafka → Flink → S3 → Athena), backups (RDS snapshots), artifatos de CI/CD, e dados históricos (via AWS Glue pra Databricks). Políticas de lifecycle movem dados antigos pra Glacier automaticamente — reduz custo em 90%.

7 Load Balancers — ALB e NLB

AWS oferece dois tipos principais de load balancer — escolher depende do caso de uso.

ALB (Application LB)NLB (Network LB)
CamadaCamada 7 (HTTP/HTTPS)Camada 4 (TCP/UDP)
InteligênciaEntende URLs, headers, hostnamesApenas IP e porta
Performance~400k RPS~100M RPS (10x mais rápido!)
Caso de usoAPIs REST, microsserviçosStreaming, gaming, IoT
LatênciaAlguns msUltra baixa (microsegundos)

Health checks — O LB fica verificando se as máquinas tão vivas (ping a cada X segundos). Se uma morrer, ele para de mandar tráfego pra ela.

Sticky sessions — Força um cliente sempre ir pro mesmo backend (útil pra sessões locais, mas quebra escalabilidade). Melhor é usar Redis compartilhado.

💡 No PicPay... usamos ALBs na frente dos microserviços (HTTP é o padrão). ALB roteia por hostname (api.picpay.com → ms-api), por path (/users → ms-users), ou por header (mobile-app → versão beta). Para casos extremos (Kafka broker, Redis), usamos NLB.

8 Auto Scaling — Escala Automática

Auto Scaling cria e deleta máquinas automaticamente conforme a carga. Quando tráfego sobe, novos EC2s são lançados. Quando cai, máquinas ociosas são deletadas.

🎪 Analogia: Se você tem uma lanchonete, na Black Friday contrata mais atendentes. Na segunda-feira vazia, demite. Auto Scaling é isso automático.

Como funciona:

  • Launch template — Define qual imagem, tipo de máquina, security group usar
  • Auto Scaling Group — Gerencia o grupo. Você define min/desired/max instâncias
  • Scaling policies — Define quando escalar:
    • Target tracking — Mantém CPU em 70% (exemplo). Se passar, escala
    • Step scaling — Regras customizadas (se CPU > 80%, adiciona 2 máquinas)
    • Scheduled — Escala em horários específicos (sabe que 17h é pico)

💡 No PicPay... Kubernetes (EKS) gerencia escaling de pods, não EC2 direto. O cluster EKS em si usa Auto Scaling Group pra nodes (máquinas EC2) — quando pods precisam de mais recurso, novos nodes são criados automaticamente.

9 CloudWatch — Monitoramento Nativo

CloudWatch coleta métricas de tudo na AWS — EC2 (CPU, disco, rede), RDS (conexões, query time), ALB (latência, 4xx/5xx). Você cria alertas baseados nessas métricas.

📊 Analogia: Se CloudWatch fosse seu corpo, CPU seria frequência cardíaca, memória seria nível de açúcar. Você monitora e avisa se algo sai do normal.

Recursos principais:

  • Metrics — Dados coletados (CPUUtilization, NetworkIn, DatabaseConnections)
  • Logs — Eventos de aplicações (logs de aplicação, syslog)
  • Alarms — Quando métrica sai do range, faz algo (envia SNS, escala auto-scaling)
  • Dashboards — Visualizações das métricas (tipo Grafana, mas nativo)

💡 No PicPay... CloudWatch é base pra alertas, mas usamos Prometheus/Grafana como principal (mais customizável). CloudWatch é backup pra alertas de componentes AWS puros (RDS, S3, ELB).

10 IAM — Identity and Access Management

IAM controla quem pode acessar o quê na AWS. Não é segurança física — é autorização (quem pode deletar um RDS? apenas admin).

🔐 Analogia: IAM é o cartão de acesso do prédio. Seu cartão abre a porta do seu andar (EC2s), mas não abre a sala do cofre (RDS production).

Componentes:

  • Users — Pessoas (dev@picpay.com, devops@picpay.com)
  • Groups — Grupos de usuários (SREs, Developers) — facilita atribuir permissões iguais
  • Roles — Atribuições. Uma role pode ser "assumida" temporariamente (EC2 assume role pra acessar S3)
  • Policies — Documentos JSON que definem permissões (allow/deny)

Exemplo de policy:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}

Princípio de menor privilégio: dê o mínimo de permissão necessário. Dev não precisa deletar RDS — precisa apenas conectar.

💡 No PicPay... cada microsserviço tem uma role IAM com permissões específicas (ms-users acessa RDS user-db e S3 avatars, mas não acessa RDS payment-db). Quando um dev precisa escalar privilégios, usa "assume role" temporária (auditado).
