Mensageria & Eventos

Kafka, SQS e comunicação assíncrona entre serviços

1 O que é Mensageria?

Mensageria é um modelo de comunicação assíncrona entre sistemas. Em vez de o Serviço A ligar direto pro Serviço B e ficar esperando a resposta (síncrono), ele deposita uma mensagem numa fila ou tópico — e o Serviço B lê quando puder.

O benefício principal é o desacoplamento: quem envia não precisa saber quem recebe, e se o receptor estiver fora do ar, a mensagem fica guardada até ele voltar.

📮 Analogia: Comunicação síncrona (REST) é um telefonema — você liga, a pessoa precisa atender agora, e vocês ficam presos na linha. Mensageria é o correio — você deposita a carta na caixa e segue sua vida. O destinatário pega quando quiser. Se ele estiver viajando, a carta espera.

Benefícios da mensageria:

  • Desacoplamento — Producer e consumer não se conhecem diretamente
  • Resiliência — Se o consumer cair, as mensagens ficam retidas
  • Escalabilidade — Múltiplos consumers podem processar em paralelo
  • Picos de carga — A fila absorve spikes, consumer processa no próprio ritmo
  • Auditoria — Mensagens ficam registradas (especialmente no Kafka)

💡 No PicPay... centenas de microserviços se comunicam via mensageria. Quando você faz um Pix, o serviço de pagamento publica um evento no Kafka — e vários consumers reagem: notificação push, atualização de extrato, contabilidade, antifraude. Tudo assíncrono, tudo desacoplado.

2 Kafka — Fundamentos

Apache Kafka é um sistema de streaming distribuído. Diferente de uma fila tradicional (onde a mensagem some depois de lida), o Kafka é um log distribuído — as mensagens ficam guardadas por dias/semanas e podem ser relidas.

📰 Analogia: Uma fila tradicional (SQS) é um bilhete: você pega, lê, e ele desaparece. Kafka é um jornal: ele fica disponível pra todo mundo ler, cada leitor marca até onde já leu (offset), e edições antigas ficam guardadas na hemeroteca.

Conceitos fundamentais:

  • Tópico (Topic) — Canal de mensagens por assunto. Ex: payments-events, user-notifications
  • Partição (Partition) — Subdivisão do tópico pra paralelismo. Cada partição é um log ordenado e imutável
  • Offset — Posição da mensagem na partição. O consumer lembra até onde leu
  • Broker — Servidor Kafka que armazena as partições. Um cluster tem vários brokers
  • Cluster — Conjunto de brokers que trabalham juntos. Alta disponibilidade por replicação
  • Retenção — Tempo que as mensagens ficam guardadas (ex: 7 dias, 30 dias)
           Tópico: payments-events
  ┌─────────────────────────────────────────┐
  │  Partição 0: [msg0][msg1][msg2][msg3]→  │
  │  Partição 1: [msg0][msg1][msg2]→        │
  │  Partição 2: [msg0][msg1][msg2][msg3]→  │
  └─────────────────────────────────────────┘
                    offset →

  Producer envia → Kafka decide a partição (por key ou round-robin)
  Consumer lê   → Cada consumer group tem seu próprio offset

💡 No PicPay... rodamos múltiplos clusters Kafka (MSK na AWS): um pra Banking, outro pra Platform, outro pra CX. Cada cluster tem dezenas de brokers. Tópicos críticos como banking-wallet_notify-movement têm retenção de dias e replicação 3x pra garantir que nenhum evento financeiro se perca.

3 Producers e Consumers

O Producer envia mensagens pro Kafka. O Consumer lê mensagens do Kafka. Parece simples, mas a forma como cada um funciona tem nuances importantes.

Producer — quem envia:

  • Serialização — Converte o objeto pra bytes (JSON, Avro, Protobuf) antes de enviar
  • Partitioning — Se enviar com key (ex: userId), todas as msgs da mesma key vão pra mesma partição (garante ordem)
  • Acks — Nível de confirmação: acks=0 (fire and forget), acks=1 (líder confirmou), acks=all (todas as réplicas confirmaram)
  • Retry — Se falhar, reenvia automaticamente (configurável)

Consumer — quem lê:

  • Desserialização — Converte bytes de volta pro objeto
  • Poll — Consumer faz polling ativo: "tem mensagem nova?" — não é push
  • Commit de offset — Marca até onde já processou (auto-commit ou manual)
  • Reprocessamento — Pode resetar o offset pra reler mensagens antigas (replay)
// Producer (pseudo-código)
producer.send("payments-events", key="user-123", value={
  "type": "PIX_SENT",
  "amount": 150.00,
  "timestamp": "2026-03-01T10:30:00Z"
});

// Consumer (pseudo-código)
while (true) {
  messages = consumer.poll(timeout=1000ms);
  for (msg in messages) {
    process(msg.value);    // processa a mensagem
    consumer.commit();     // marca como lida
  }
}

💡 No PicPay... usamos serialização JSON na maioria dos tópicos (flexibilidade) e Avro em tópicos de alta performance (schema registry garante compatibilidade). O acks=all é padrão pra tópicos financeiros — não podemos perder um evento de pagamento.

4 Consumer Groups

Um Consumer Group é um conjunto de consumers que trabalham juntos pra processar um tópico. O Kafka distribui as partições entre os consumers do grupo — cada partição vai pra exatamente um consumer.

🍕 Analogia: Imagine uma pizza com 6 fatias (partições) e 3 amigos (consumers no grupo). Cada amigo pega 2 fatias. Se um amigo for embora, os 2 restantes redistribuem — cada um pega 3. Se chegar um 4º amigo, redistribui de novo. Mas nunca 2 amigos pegam a mesma fatia ao mesmo tempo.

Regras fundamentais:

  • 1 partição → 1 consumer — Uma partição nunca é lida por 2 consumers do mesmo grupo simultaneamente
  • Mais consumers que partições? — Os extras ficam ociosos (sem partição pra ler)
  • Rebalanceamento — Quando um consumer entra ou sai do grupo, Kafka redistribui as partições automaticamente
  • Grupos independentes — Dois groups diferentes lêem as MESMAS mensagens (cada group tem seu offset)
  Tópico: payments-events (6 partições)

  Consumer Group A (processamento):       Consumer Group B (auditoria):
  ┌─────────┐ ┌─────────┐ ┌─────────┐   ┌─────────────┐
  │Consumer 1│ │Consumer 2│ │Consumer 3│   │ Auditor (1) │
  │ P0 + P1  │ │ P2 + P3  │ │ P4 + P5  │   │ P0..P5 (all)│
  └─────────┘ └─────────┘ └─────────┘   └─────────────┘
  → Cada um lê 2 partições               → 1 consumer lê tudo

💡 No PicPay... cada microserviço consumer tem seu próprio group ID (ex: ms-statement-persister-group). Assim, o serviço de extrato e o serviço de notificação lêem as mesmas mensagens de pagamento de forma independente. Se o extrato atrasar, a notificação não é afetada.

5 SQS da AWS — Filas Gerenciadas

Amazon SQS (Simple Queue Service) é uma fila de mensagens totalmente gerenciada. Diferente do Kafka (que você opera), o SQS é serverless — AWS cuida de tudo.

Dois tipos de fila:

AspectoStandardFIFO
OrdemBest-effort (pode chegar fora de ordem)Garantida (First In, First Out)
DuplicatasPossível (at-least-once)Exatamente uma vez (dedup por ID)
ThroughputIlimitado300 msg/s (3.000 com batching)
Uso idealNotificações, jobs assíncronosProcessamento financeiro, onde ordem importa

Conceitos-chave do SQS:

  • Visibility Timeout — Quando um consumer lê uma msg, ela fica "invisível" pra outros consumers durante X segundos. Se o consumer não deletar a tempo, a msg reaparece e outro pode pegar
  • Long Polling — Consumer pergunta "tem msg?" e SQS espera até 20s antes de responder vazio. Reduz chamadas e custo
  • Retention — Mensagens ficam na fila por até 14 dias (padrão: 4 dias)
  • Max Message Size — 256 KB por mensagem. Pra payloads grandes, usar S3 + referência

🎫 Analogia: SQS Standard é a fila do banco — geralmente a ordem é respeitada, mas às vezes alguém fura. SQS FIFO é a fila com senha numerada — ordem rigidamente garantida, ninguém fura, mas a fila anda mais devagar.

💡 No PicPay... usamos SQS pra tarefas que não precisam do peso do Kafka: processamento de statements (extrato), envio de notificações em lote, jobs de reconciliação. O ms-statement-persister consome de 10+ filas SQS em paralelo, cada uma com sua DLQ.

6 Dead Letter Queue (DLQ)

DLQ é uma fila especial pra mensagens que falharam. Quando um consumer tenta processar uma mensagem e falha N vezes seguidas, em vez de ficar tentando pra sempre (e travando a fila), a mensagem é movida pra DLQ.

📬 Analogia: Imagine o carteiro tentando entregar uma carta 3 vezes. Na 1ª, ninguém em casa. Na 2ª, endereço errado no interfone. Na 3ª, porteiro recusa. Depois de 3 tentativas, a carta vai pro "setor de cartas não entregues" (DLQ) em vez de ficar no caminhão pra sempre.

Como funciona:

  • maxReceiveCount — Quantas vezes uma msg pode ser lida sem ser deletada antes de ir pra DLQ (ex: 3 tentativas)
  • Fila source → aponta pra fila DLQ como redrive policy
  • Msgs na DLQ — Precisam ser investigadas manualmente (ou com automação)
  • Reprocessamento — Após corrigir o bug, mover msgs da DLQ de volta pra fila original
  Fila Principal                  DLQ
  ┌──────────────┐              ┌──────────────┐
  │  [msg A] ────│──(3 falhas)──│→ [msg A]     │
  │  [msg B]     │              │  [msg C]     │
  │  [msg D]     │              │  (investigar)│
  └──────────────┘              └──────────────┘

  Config: maxReceiveCount = 3
  → Após 3 tentativas de consumo sem delete, msg vai pra DLQ

Boas práticas:

  • Alertar quando DLQ tiver msgs — Se tem msg lá, algo deu errado
  • Retenção da DLQ ≥ fila original — Senão a msg pode expirar antes de você investigar
  • Monitorar idade das msgs na DLQ — Msgs velhas = problema negligenciado

💡 No PicPay... toda fila SQS tem uma DLQ associada. Quando msgs chegam na DLQ, um alerta dispara no Slack do time dono. O time investiga, corrige o bug, e usa o "redrive" pra reprocessar as mensagens. Kafka também tem o conceito: tópicos *.DLQ ou *.retry pra msgs que falharam no consumer.

7 Kafka vs SQS — Quando usar cada um

Não é que um seja melhor que o outro — cada um resolve problemas diferentes. A escolha depende do cenário.

AspectoKafkaSQS
ModeloLog distribuído (pub/sub)Fila de mensagens (point-to-point)
RetençãoConfigurável (dias/semanas)Até 14 dias
ReplaySim (resetar offset)Não (msg deletada após consumo)
Multi-consumerSim (cada group lê tudo)Não (msg vai pra 1 consumer)
OrdemPor partiçãoFIFO ou best-effort
ThroughputMilhões msg/sIlimitado (standard) / 300 msg/s (FIFO)
OperaçãoGerenciar cluster (MSK)Serverless (zero ops)
CustoCluster fixo (brokers 24/7)Pay-per-use (por request)
Caso idealEvent streaming, múltiplos consumers, replayJobs async, filas simples, desacoplamento leve

📺 Analogia: Kafka é a Netflix — o conteúdo fica disponível, vários perfis (consumers) assistem o mesmo filme, e você pode reassistir quando quiser. SQS é a padaria com senha — quando chamam seu número, você pega o pão, e o número não serve mais pra ninguém.

💡 No PicPay... a regra geral é: Kafka pra eventos de domínio (pagamento feito, saldo atualizado, cartão emitido) onde múltiplos serviços precisam reagir. SQS pra jobs internos (persistir extrato, enviar batch de notificações) onde só um consumer processa. Se precisa de replay → Kafka. Se é fire-and-forget → SQS.

8 Garantias de Entrega

Em sistemas distribuídos, entregar uma mensagem exatamente uma vez é extremamente difícil. Existem 3 níveis de garantia:

GarantiaDescriçãoRisco
At-most-onceEntrega no máximo 1 vez. Pode perder.Perda de mensagem
At-least-onceEntrega pelo menos 1 vez. Pode duplicar.Mensagem duplicada
Exactly-onceEntrega exatamente 1 vez. O santo graal.Complexidade alta, custo de performance

📧 Analogia: At-most-once = enviar e-mail sem confirmação de leitura (pode não chegar, tudo bem). At-least-once = enviar com "acusar recebimento", reenviando se não receber confirmação (pode chegar 2x). Exactly-once = carta registrada com AR (garantia total, mas custa mais caro e é mais lento).

Idempotência — a solução prática:

Na prática, a maioria dos sistemas usa at-least-once + idempotência. O consumer é preparado pra receber a mesma mensagem 2x e produzir o mesmo resultado:

// Consumer idempotente (pseudo-código)
function processPayment(msg) {
  // Checa se já processou (por chave única)
  if (database.exists(msg.transactionId)) {
    log("Já processado, ignorando duplicata");
    return;  // não processa de novo
  }
  
  // Processa e marca como feito
  database.save(msg.transactionId, result);
  commit();
}
  • Kafka — Padrão é at-least-once. Suporta exactly-once com transactional API (enable.idempotence=true)
  • SQS Standard — At-least-once (pode duplicar)
  • SQS FIFO — Exactly-once (por deduplication ID dentro de uma janela de 5 min)

💡 No PicPay... todo consumer de evento financeiro é idempotente. Usamos chaves compostas (userId + walletName + originId + codeId) armazenadas no Redis com SET NX pra impedir processamento duplicado. Se o Kafka re-entregar uma mensagem de débito, o consumer detecta que já processou e ignora.

9 Monitoramento de Filas

Filas e tópicos sem monitoramento são bomba-relógio. Os sinais de alerta mais importantes:

MétricaO que medeAlerta quando...
Consumer LagDiferença entre último offset produzido e último consumidoLag cresce continuamente (consumer não acompanha)
ThroughputMsgs/s produzidas e consumidasProdução sobe mas consumo não acompanha
DLQ SizeQuantidade de msgs na DLQQualquer msg na DLQ (= falha de processamento)
Consumer HealthSe os consumers estão vivos e respondendoConsumer parou de fazer poll (heartbeat timeout)
Age of MessageTempo que a msg mais antiga está na filaMsg com mais de X minutos sem ser processada

🚦 Analogia: Consumer lag é como trânsito: se a entrada de carros (producers) é maior que a saída (consumers), forma congestionamento. Monitorar lag é colocar câmera no trânsito — se o congestionamento cresce, você abre mais faixas (escala consumers).

Ferramentas comuns:

  • Prometheus + Grafana — Métricas de consumer lag, throughput, erro rate
  • CloudWatch (SQS) — ApproximateNumberOfMessages, ApproximateAgeOfOldestMessage
  • Kafka Manager / AKHQ — UI pra visualizar tópicos, partições, consumer groups
  • KEDA — Auto-scaling de consumers baseado em lag ou tamanho da fila
# PromQL — Consumer lag por group
kafka_consumer_group_lag{
  group="ms-statement-persister",
  topic="wallet-movement-events"
}

# CloudWatch — Msgs visíveis na fila SQS
ApproximateNumberOfMessagesVisible > 1000 por 5 minutos → ALERTA

💡 No PicPay... todo consumer Kafka tem dashboard no Grafana mostrando lag em tempo real. Se o lag passa de um threshold, o KEDA escala automaticamente mais pods consumers. Pra SQS, alertas CloudWatch disparam quando ApproximateAgeOfOldestMessage passa de 5 minutos — sinal de que o processamento travou.

10 Padrões de Design com Mensageria

Mensageria não é só "enviar e receber". Existem padrões arquiteturais poderosos construídos em cima dela:

1. Event Sourcing

Em vez de salvar o estado atual no banco, você salva todos os eventos que aconteceram. O estado é reconstruído reprocessando os eventos.

  • Ex: Conta bancária não salva "saldo = R$500". Salva: +R$1000 (depósito), -R$300 (Pix), -R$200 (compra) → saldo calculado = R$500
  • Vantagem: Auditoria completa, possibilidade de replay

2. CQRS (Command Query Responsibility Segregation)

Separa o modelo de escrita (commands) do modelo de leitura (queries). Escritas vão pro banco principal, leituras vêm de uma view otimizada (materializada via eventos).

  • Vantagem: Escala leitura e escrita independentemente

3. Saga Pattern

Orquestra transações distribuídas via eventos. Cada serviço faz sua parte e publica um evento. Se alguém falhar, eventos de compensação desfazem as etapas anteriores.

  • Ex: Pagamento → Reserva estoque → Gera nota fiscal. Se a nota fiscal falhar → evento "cancelar reserva" → evento "estornar pagamento"

4. Outbox Pattern

Garante consistência entre banco e mensageria. Em vez de salvar no banco E publicar no Kafka (2 operações que podem falhar separadamente), você salva o evento numa tabela "outbox" no MESMO banco (transação atômica). Um processo separado lê a outbox e publica no Kafka.

  Outbox Pattern:
  ┌────────────────────────────┐      ┌─────────┐
  │ Banco de Dados             │      │  Kafka   │
  │ ┌──────────┐ ┌──────────┐ │      │          │
  │ │ tabela   │ │ outbox   │ │──────│→ tópico  │
  │ │ principal│ │ (eventos)│ │ poll │          │
  │ └──────────┘ └──────────┘ │      └─────────┘
  └────────────────────────────┘
  1 transação atômica (ACID)    publisher lê outbox

🧱 Analogia: Outbox é como o protocolo de despacho dos Correios. Você não joga a carta direto na rua (pode perder). Você coloca na caixa de saída do escritório (outbox). O carteiro passa periodicamente, coleta tudo da caixa e garante a entrega.

💡 No PicPay... o ms-wallets-investment-adapter usa Outbox Pattern: ao processar um investimento, salva a operação + o evento na mesma transação do PostgreSQL. Um publisher separado lê a tabela outbox e publica no Kafka — garantindo que nenhum evento financeiro se perca mesmo se o Kafka estiver temporariamente indisponível.
