DevOps Básico

Semana 3 — A bridge entre desenvolvimento e produção: pipelines, containers e automação

1 O que é DevOps?

DevOps não é um cargo. É uma filosofia e conjunto de práticas que aproxima o time de desenvolvimento e o time de operações. Em vez de "devs" jogarem código pela cerca e "ops" ficarem resolvendo incidentes, todos trabalham juntos.

🏭 Analogia: Imagine uma fábrica de carros. "Dev" desenha o carro, "Ops" monta. DevOps é ambos trabalhando no mesmo chão, otimizando o processo de produção juntos.

Os pilares do DevOps:

  • Automação — Scripts, pipelines, infraestrutura como código
  • Colaboração — Devs e Ops conversando desde o dia zero
  • Integração contínua — Código é testado automaticamente a cada push
  • Entrega contínua — Deploy pra produção é automático e seguro
  • Observabilidade — Saber o que está acontecendo em produção (logs, métricas, traces)

DevOps reduz o tempo entre uma ideia e ela estar rodando em produção — de meses para horas ou minutos.

💡 No PicPay... o Command Center é um exemplo vivo de DevOps. Devs fazem deploy direto, monitoram seus próprios serviços, e resolvem incidentes colaborando com ops. Sem DevOps, cada deploy levaria dias e dependeria de outros times.

2 Containers e Docker

Antes de containers, a vida era um caos: "funciona na minha máquina, mas em produção não funciona". Por quê? Porque dev tinha Python 3.8 e produção tinha 3.7. Dev tinha libpq na máquina e produção não.

Containers resolvem isso: empacotam tudo — código, dependências, runtime, configurações — num arquivo único. Roda igual em qualquer lugar.

📦 Analogia: Containers são como ships (contêineres marítimos). Você coloca a mercadoria dentro do container, fecha, e ele vai de navio, trem ou caminhão — sempre igual. Ninguém mexe no que tá dentro.

Docker é o padrão pra criar e rodar containers:

FROM python:3.11
RUN pip install flask
COPY . /app
WORKDIR /app
CMD ["python", "app.py"]

Esse Dockerfile descreve: começa com Python 3.11 oficial, instala Flask, copia o código, e roda a app. Resultado: uma imagem que sempre se comporta igual em qualquer lugar.

Imagem vs Container:

  • Imagem — O "molde", o arquivo congelado (type)
  • Container — Uma instância rodando dessa imagem (instance)

💡 No PicPay... todos os nossos microserviços rodam em containers Docker. Quando você faz deploy, o Docker constrói a imagem, testa, e sobe no Kubernetes. Isso garante que a dev, staging e produção são idênticas.

3 CI/CD — Integração e Entrega Contínua

Imagina: você faz 1000 linhas de código, testa sozinho, aí quando sobe em produção descobre que quebrou outra coisa. Incidente em prod é caro.

CI (Continuous Integration) = toda vez que você faz push, um script automático roda os testes. Se algum falhar, você é avisado agora, não em produção.

CD (Continuous Delivery/Deployment) = se todos os testes passarem, o código é automaticamente colocado em produção (ou preparado pra ser).

🏃 Analogia: Sem CI/CD, cada deploy é uma cirurgia a céu aberto — leva tempo, arriscado, precisa de calma. Com CI/CD, é como um pit stop de Fórmula 1 — automático, rápido, rotineiro.

Fluxo típico:

1. Dev faz commit
2. GitHub Actions / GitLab CI dispara
3. Build da imagem Docker
4. Roda testes (unit, integration, security)
5. Se passou: push pra registry (Docker Hub, ECR)
6. Deploy no Kubernetes (automático ou com aprovação)
7. Rollback automático se métricas piorarem

💡 No PicPay... usamos GitHub Actions pra CI/CD. Cada PR é testada automaticamente. Merge pra main = deploy automático em produção (via ArgoCD, que vamos ver). Se as métricas do Grafana indicarem problema, rollback automático.

4 Infraestrutura como Código (IaC)

Antigamente: ops criava um servidor, clicava em menus, configurava manualmente. Se precisava de 10 servidores iguais? Pior: tinha que fazer 10 vezes à mão.

Com IaC, você descreve a infra num arquivo — assim como código. Depois? Um comando cria tudo automaticamente. Mudou o requisito? Edita o arquivo, roda de novo.

🏗️ Analogia: IaC é a planta de uma casa. Arquiteto desenha, constrói uma, depois constrói cem idênticas. Sem planta, cada casa seria diferente.

Ferramentas comuns:

  • Terraform — Infraestrutura na AWS, GCP, Azure (o mais popular)
  • CloudFormation — Nativa da AWS, menos portável
  • Ansible — Configuração de máquinas (playbooks)
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  tags = {
    Name = "web-server-prod"
  }
}

Esse arquivo Terraform cria uma instância EC2 na AWS. Rodou uma vez? Pode rodar de novo — cria exatamente a mesma. Perfeito pra disaster recovery.

💡 No PicPay... usamos Terraform pra toda infraestrutura AWS — VPCs, Security Groups, RDS, Lambda. Tudo versionado no Git. Quando alguém precisa de um recurso novo, edita o .tf, faz PR, e after aprovação roda terraform apply. Nenhuma clique manual.

5 Monitoramento e Alertas

Produção está rodando, mas você não sabe o que está acontecendo. Usuários reclamam que é lento, mas você não vê nada nos logs. Que desastre.

Monitoramento = visibilidade. Você coleta métricas (CPU, memória, requisições por segundo, tempo de resposta) e cria dashboards. Se algo sai do normal? Um alerta te avisa.

📊 Analogia: Monitoramento é o painel de controle do carro. Sem ele, você não sabe se está com combustível baixo até o carro parar. Com ele, ve o indicador e abastece antes.

Pilares de monitoramento:

  • Métricas — Números (CPU%, requisições/s, latência P99)
  • Logs — Eventos, erros, traces de execução
  • Traces distribuídos — Ver uma requisição passar por vários serviços
  • Alertas — Quando uma métrica ultrapassa threshold (ex: CPU > 80% por 5 min)

💡 No PicPay... usamos Prometheus (métricas), OpenSearch (logs) e Jaeger (traces). Grafana consome tudo e cria dashboards. CloudWatch Alarms monitora e notifica. Se um serviço cai, você sabe em segundos — não quando o usuário reclama.

6 Versionamento com Git

Git é a coluna vertebral de qualquer projeto DevOps. Tudo fica no Git: código, IaC, configurações. Histórico completo, possibilidade de voltar, colaboração entre times.

Branching strategy — GitFlow:

  • main — Produção. Código aqui sempre está deployado. Protegido: só merge via PR aprovada.
  • develop — Versão de desenvolvimento. Features se juntam aqui antes de ir pra main.
  • feature/nome — Branches de feature. Cada dev trabalha isolado, depois abre PR.
  • hotfix/nome — Hotfixes pra produção. Críticos, direto da main.

Fluxo típico:

git checkout -b feature/new-api
# faz mudanças, commits
git push origin feature/new-api
# abre PR no GitHub → code review → merge
# CI/CD dispara automaticamente

💡 No PicPay... tudo passa por PR — código, IaC, documentação. PRs exigem aprovação + testes passando. Histórico de commits é auditável (compliance). Sem histórico em Git, você não sabe quem mudou o quê e por quê.

7 Secrets Management

API keys, senhas de banco de dados, tokens OAuth — esses dados sensíveis NUNCA podem estar no Git. Se alguém clonar o repo, não pode ter acesso a eles.

A forma correta: armazenar secrets num cofre seguro e o app busca lá em runtime.

🔐 Analogia: Secrets são como as chaves da sua casa. Você não as deixa na porta. Guarda num lugar seguro — um cofre, uma agência de coisas seguras. Só você e quem você confia têm acesso.

Ferramentas comuns:

  • AWS Secrets Manager — Cofre nativo da AWS, rotação automática
  • HashiCorp Vault — Multi-cloud, auditoria completa
  • Kubernetes Secrets — Pra dados em K8s (básico, nem sempre recomendado em prod)

Regra de ouro: se você consegue ver a senha no terminal, ela não deveria estar lá. Deve estar criptografada em repouso e só descriptografada quando usada.

💡 No PicPay... todos os secrets estão em AWS Secrets Manager ou Vault. No Dockerfile ou código, não tem NENHUMA senha hardcoded. O app lê de lá. Se um dev vaza um token no GitHub? Dentro de horas (graças ao scanning), a gente rotaciona.

8 Disaster Recovery e Backup

Um data center queima. Disco falha. Alguém deleta tabela errada do banco. Desastres acontecem. Você precisa estar preparado pra voltar.

Conceitos importantes:

  • RTO (Recovery Time Objective) — Quanto tempo pode ficar fora do ar? (ex: 1 hora)
  • RPO (Recovery Point Objective) — Quantos dados você pode perder? (ex: últimos 5 minutos)
  • Backup — Cópia dos dados num lugar seguro, longe de produção
  • Réplica — Cópia ativa em outra região/data center

Estratégia de backup típica:

  • Backup completo 1x por semana (domingo) → S3 / Glacier
  • Backup incremental diário (Seg-Sáb) → S3
  • Replicação em tempo real pra outra região (RDS, MongoDB)

💡 No PicPay... RDS tem replicação multi-AZ (outro data center na mesma região) e backups automatizados. MongoDB tem replicação cross-region. S3 tem versionamento. Se data center inteiro cai, a gente sobe em outro em minutos — não horas.
